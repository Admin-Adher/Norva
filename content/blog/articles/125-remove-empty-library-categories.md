# How to Find and Remove Empty Library Categories

> **In short:** Clear search and filters, check the relevant source, include unavailable or archived states where appropriate, and verify the category on another supported view before calling it empty. Document why it exists and whether rules depend on it. Remove only the category label or rule through supported controls, then retest navigation and rollback.

An empty category can be genuinely obsolete, temporarily empty, filtered out, source-defined, or reserved by an active plan. The correct action depends on which state applies.

## Build a candidate list

From [the pre-cleanup audit](/blog/audit-library-before-cleanup/), list categories with:

- zero visible items;
- no items after recent reclassification;
- duplicate or overlapping labels;
- no clear household meaning;
- no use during the last review period;
- source removed or changed;
- planned future use but no current members.

Do not delete while collecting candidates. Record exact label, source, displayed count, device, filters, and audit time.

## Confirm “empty” without filters

For each candidate:

1. clear search text;
2. clear source, availability, year, rating, language, and subtitle filters;
3. return to the broadest relevant library view;
4. open the category;
5. check whether unavailable or archived items are intentionally hidden;
6. refresh through a supported route;
7. compare another supported device or browser view when relevant.

If items appear after clearing a filter, the category is not empty. The issue is filter context or category visibility.

## Determine who defines the category

A category may be:

- supplied by the connected source;
- created by a household rule;
- generated by a migration mapping;
- retained as a reserved category for an approved future plan;
- left behind after reclassification.

Do not edit source-defined structure through an unsupported route. Document whether the proposed action changes only a local catalogue label, a mapping, or source data.

Norva organises compatible authorised sources; it cannot guarantee that a removed source category will not reappear after the source refreshes.

## Check dependencies

Search the collection plan, metadata rulebook, migration ledger, and decision log for the category label. Ask:

- Does intake assign new items here?
- Does another category mapping point to it?
- Is it a parent for subcategories?
- Is a household findability task built around it?
- Is it an archive return destination?
- Is a pending import expected to use it?
- Did a previous decision deliberately preserve it?

Remove or update dependencies before deleting the label.

## Classify the candidate

Use five states:

- **Filtered empty:** items exist but current filters hide them.
- **Source empty:** source currently exposes no members.
- **Temporarily empty:** approved future intake or migration will use it soon.
- **Obsolete:** no members, dependencies, or approved future purpose.
- **Unknown:** evidence remains incomplete.

Only “obsolete” is ready for removal. Unknown belongs in review.

## Protect rollback and scope

Record the category label, source, rule, dependencies, and previous count. Verify the supported way to recreate it or restore the relevant catalogue state. Use [the reversible cleanup guide](/blog/make-catalog-cleanup-reversible/) before removing several categories.

The action should remove the empty category structure, not media items. Read confirmation text carefully. If the interface combines both actions, stop and seek current documentation or support.

## Reconcile after removal

After every small batch:

- compare category counts with the baseline;
- search for representative items;
- inspect uncategorised items;
- verify source refresh behaviour;
- update mappings and intake rules;
- record the decision;
- repeat on another supported view if the category was cross-device.

Feed newly uncategorised items into [the triage method](/blog/triage-uncategorized-media/) and keep the batch in [the complete cleanup ledger](/blog/catalog-cleanup-master-plan/).

## Common mistakes and limitations

- Judging emptiness with filters active.
- Removing a source-defined label as if it were local.
- Ignoring unavailable or archived states.
- Deleting a category with active intake mappings.
- Using category removal to delete media.
- Processing many labels without a pilot.
- Assuming a removed source category cannot return.
- Treating a reserved planning category as permanent without a review date.

Category controls vary by source and application. Verify the current supported action and its scope.

## Frequently asked questions

### Can a category be empty on one device only?

Yes, when filters, refresh state, source selection, or profile context differ. Compare the same account, source, filters, and category before removal.

### Should a planned future category remain visible?

Only when its near-term purpose, owner, and review date are approved. Otherwise, keep the plan outside the live catalogue until items exist.

### Will removing a category delete its media?

It should not be assumed. Read the exact confirmation and current documentation. Proceed only when the action's scope is understood and recovery is available.
