# Runtime Differs Between Versions? Why Identity Matters

> **In short:** Treat runtime as an identity clue, not proof that one version is wrong. Record source label, title code, year, media type, edition, season or episode, language cues, displayed duration and format, playback boundaries when authorized and necessary, device, application version, and timestamp. Compare each version independently, preserve progress, and avoid converting rough values into false precision or assuming how Norva groups alternate cuts.

Two versions can differ because they are alternate cuts, regional editions, episodes, source records, or files with different opening and ending material. Runtime helps distinguish them but does not explain the difference by itself.

Use one comparison unit throughout: either the same source version across views or two clearly different versions in one view. Mixing those questions can make a display-format difference look like a content-length difference and weakens every later conclusion.

## Confirm separate version identity

Record source, title, year, media type, edition, artwork, language, season, episode, and grouped-version state. If the cards are not clearly separate versions, preserve that uncertainty before comparing duration.

The [category and metadata handbook](/blog/category-metadata-troubleshooting-handbook/) provides the identity matrix.

## Record duration exactly as shown

Capture hours, minutes, seconds, percentage, or rounded card label in its original format. Do not convert “about two hours” into exact seconds or compare a rounded card value with an exact player value as if they have equal precision.

## Compare source duration fields

Through each provider's official authorized route, record the labeled duration for the matching version and timestamp. Do not assume the source field measures the same boundaries or has the same precision as every Norva view.

## Compare Norva views

Record duration on category card, search, detail page, episode list, player, or grouped-version view only where shown. A rounded list value and detailed player value are separate presentation observations.

## Inspect boundaries only when necessary

If authorized playback is available and the difference matters, note whether credits, introductions, recaps, previews, black frames, or bonus material appear at the boundaries. Do not copy media or bypass source restrictions. Boundary observations can explain content scope without defining metadata rules.

## Protect progress

Do not seek repeatedly, play several versions deeply, mark complete, or reset progress merely to measure duration. Record existing progress first. The [progress identity guide](/blog/progress-reset-after-item-identity-change/) protects household state when versions change.

## Compare year and edition

A different year may indicate a restored, extended, anniversary, or regional release. Use the [release-year verification](/blog/incorrect-release-year-metadata/) to keep edition and year fields distinct.

## Freeze view context

Record account, profile, enabled sources, filters, grouping, device, operating system, application version, and timestamp. A grouped card can switch visible versions when source or filter context changes.

## Compare another supported device

Use the same account, profile, source version, filters, grouping, and close time. Record both application versions. A duration shown differently on one device is cross-screen evidence, not proof of file analysis or local storage behavior.

## Avoid forced regrouping or edits

Do not rename files, edit duration metadata, merge cards, delete a version, repeat imports, or clear application data before saving evidence. If versions appear under the wrong title, use the [grouping identity guide](/blog/grouped-versions-wrong-title/).

## Classify the difference

Use alternate edition, different episode, different source record, rounded-versus-detailed display, different playback boundaries, language or regional version, device-specific presentation, source field changed, or unknown. Never declare one version corrupt from runtime alone.

## Common mistakes and limitations

- Matching versions by title alone.
- Comparing rounded and exact values as equal precision.
- Ignoring credits, recaps, or edition boundaries.
- Changing progress while measuring versions.
- Editing duration metadata before preserving evidence.
- Calling a duration difference corruption without proof.

## Frequently asked questions

### Does a longer runtime mean a better or complete version?

No. It shows a duration difference. Edition, source, boundaries, and authoritative version information still need comparison.

### Should card and player durations match exactly?

Do not assume equal precision or presentation. Record each labeled value and current official guidance.

### Can runtime identify the correct grouped title?

It is one identity cue, not a unique identifier. Combine it with source, year, edition, media type, and language evidence.
