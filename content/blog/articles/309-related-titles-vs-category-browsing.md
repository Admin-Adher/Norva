# Related Titles or Category Browsing: Compare the Results

> **In short:** Related-title suggestions are strongest when you want to preserve a relationship to a trusted seed. Category browsing is strongest when you want to compare a defined range. Test both with the same brief, source context, candidate cap, and readiness checks. Choose the route that produces explainable, usable candidates—not simply the longest or most novel list.

The routes answer different questions. A fair comparison controls the task instead of asking which interface looks more impressive.

## Compare their natural strengths

| Criterion | Related titles | Category browsing |
|---|---|---|
| Starting point | Verified seed | Defined category or filter set |
| Primary value | Relationship depth | Range coverage |
| Common risk | Repetitive loop | Broad, shallow scanning |
| Best evidence | Shared and differing metadata | Category membership and filters |
| Stop rule | Relationship no longer serves brief | Candidate cap or range exhausted |

Neither route guarantees relevance. Both depend on the connected source, media availability, and metadata.

## Write one matched brief

Use a brief both routes can answer: “Find a work from the same period as Seed A but outside its genre, with the required subtitles.” The seed supports the related route; period and genre filters support browsing.

Do not change criteria mid-test. If the brief evolves, record the change and start a new comparison.

## Run the related-title route

Confirm the seed’s exact identity and version. Inspect up to five suggestions without playing them. For each, record visible relation, meaningful difference, availability, language, subtitles, and decision.

Use [the route-selection guide](/blog/recommendations-search-or-browse/) if the seed itself is uncertain. Do not infer relationship strength from card order.

## Run the category route

Clear all filters, then apply the category, period, media type, or language constraints available in the current interface. Record the active filter state. Inspect the same maximum number of candidates and apply identical readiness checks.

DCMI terms for subject, date, type, format, and language help define the comparison. Library of Congress genre/form vocabularies illustrate why category labels should be explicit rather than improvised.

## Score evidence without false precision

Classify each candidate:

- **Explainable:** visible fields show why it belongs.
- **Brief fit:** it preserves and changes the intended attributes.
- **Ready:** its exact version and access options suit the viewer.
- **Novel:** it adds a direction not already represented.

Count categories only to compare this paired session; do not publish the result as a general performance metric. A route can produce fewer candidates but a better shortlist.

If the goal is genre expansion, apply [the two-axis broadening map](/blog/broaden-discovery-beyond-one-genre/) to the strongest candidates.

## Choose a route by scenario

Use related titles when relationship continuity matters and the seed is trustworthy. Use category browsing when coverage, contrast, or a specific facet matters. Combine them when one route supplies a seed and the other validates variety.

Turn the final set into [a focused recommendation shortlist](/blog/turn-recommendations-into-shortlist/) rather than saving every candidate permanently.

Record any overlap between routes as a separate observation. Overlap can strengthen discoverability evidence, but the candidate still needs the same version and access checks.

Norva’s public features include source-based recommendations and library organisation. Exact filters, categories, and ordering must be verified in the current product before publication.

## Original evidence: paired-route sheet

Create two five-row tables with identical columns: Identity, Relationship or Category Evidence, Brief Fit, Version Ready, Decision. Hide the route labels and ask another reviewer which set better answers the brief and why.

The sheet supports a scenario-specific choice. It cannot declare one route universally superior or compare internal implementations.

## Common mistakes and limitations

- Giving one route more candidates or time.
- Changing the brief between routes.
- Leaving hidden filters active.
- Using card order as relevance evidence.
- Ignoring version and subtitle readiness.
- Treating source absence as route failure.

## Frequently asked questions

### Can related titles and categories return the same candidate?

Yes. Record that overlap; it can strengthen explainability but does not automatically make the candidate ready.

### Is category browsing more diverse?

Not always. Diversity depends on the selected range, source coverage, metadata, and brief.

### Should I test more than five candidates?

Use the smallest equal cap that supports the decision. Five is a worksheet example, not a universal limit.
