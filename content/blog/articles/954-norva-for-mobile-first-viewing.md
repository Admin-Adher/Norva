# A Norva Evaluation Guide for Mobile-First Viewing

> **In short:** Confirm that the intended mobile device is supported, then test Norva with the touch gestures, interruptions, network contexts, and viewing duration you actually expect. Use known media from one authorized source, inspect available tracks, storage pressure, privacy, and return behavior. Treat offline access as conditional on device, source, and rights. Add a second screen only if continuity is genuinely important.

Mobile-first viewing is personal, portable, and frequently interrupted. That makes the decision different from a scaled-down TV evaluation. The test should reflect one-handed navigation, changing attention, local storage, and the privacy of a device that may travel everywhere.

## Describe the real mobile workflow

List the three tasks you perform most: finding an item, short playback, long playback, resuming after interruption, or viewing away from the usual network. Mark which are essential and which are conveniences.

Use the [Norva evaluation framework](/blog/norva-evaluation-framework/) for the full decision, then weight these mobile tasks separately.

## Confirm the supported device context

Check current official Norva information for support. Record device category, operating-system version, application version, input method, available storage before the test, and date. Do not infer support from a different model or assume every store listing and region behaves identically.

The [first-screen guide](/blog/choose-first-norva-screen/) helps decide whether account or source entry is better on another private screen.

## Trace touch navigation

With one profile and one known item, browse, search, open details, play, pause, seek, inspect available tracks, exit, and return. Record unclear tap targets, accidental gestures, overlays, and the path back. Repeat once without changing source or network context.

Do not judge navigation while rapidly tapping through a stalled screen. Record the first unexpected response and wait for an observable state.

## Test controlled interruptions

Choose interruptions that match your use, such as briefly switching applications, locking and unlocking the screen, or receiving a permitted system prompt. Run only one per sample. Observe playback, audio, controls, profile state, and return position without assuming a particular background policy.

Repeat the normal-exit route separately so interruption behavior is not confused with ordinary resume.

## Compare relevant connectivity contexts

If you expect viewing on more than one network, run the same short authorized sample in each permitted context. Record network category and time, not a precise location or network password. A failure may involve device, source reachability, rights, or network conditions; do not label it automatically as a Norva defect.

Frequent travelers can combine this with the [travel evaluation guide](/blog/norva-for-frequent-travel/) while respecting source terms and local rules.

## Evaluate storage and offline needs

Record device free space before and after any legitimate offline test. Offline access exists only when the device, source, and associated rights permit it. Do not assume every item can be stored or that availability continues indefinitely.

First define the need with the [offline-needs evaluation](/blog/evaluate-offline-viewing-needs-norva/), then plan recurring review through the [mobile storage audit](/blog/mobile-media-storage-audit/).

## Check privacy on a personal device

Consider lock-screen visibility, screenshots, screen recording, clipboard tools, shared device access, backups, and repair handoff. Use neutral identifiers in the evaluation log. Read the current privacy policy for Norva's data practices, while remembering that device-level behavior is a separate layer.

Never store source credentials in an ordinary note for convenience.

## Verify accessibility and tracks

Use media with known audio and subtitle tracks. Check discoverability, selection, readability at the normal handheld distance, and preference behavior where available. Languages and subtitles depend on the source and media; a profile preference cannot create a missing track.

Include orientation or text-size contexts only if they are part of the real workflow, and report observed behavior without inventing product controls.

## Decide whether another screen matters

If the household also uses TV or Web, create a distinctive progress position and compare it on one supported secondary device. Keep account, profile, source, item, and timestamps aligned. The [cross-screen continuity method](/blog/evaluate-norva-cross-screen-continuity/) prevents a loose comparison.

Do not penalize a mobile-first plan for an unused screen, but do not ignore an essential handoff.

## Common mistakes and limitations

- Assuming support from a similar mobile device.
- Mixing interruptions, network changes, and settings in one run.
- Treating conditional offline access as universal downloading.
- Sharing screenshots with account or source details.
- Testing unknown media tracks.
- Ignoring storage and device privacy after the trial.

## Frequently asked questions

### Does mobile-first require offline viewing?

No. Define the actual network and travel needs first. Offline access is relevant only when the device, source, and rights support it.

### Should I test on every phone in the household?

No. Test the primary supported device and one materially different essential context, if any.

### Can a TV result stand in for mobile evaluation?

No. Touch, interruption, storage, viewing distance, and local privacy context differ, so test the real mobile workflow.
