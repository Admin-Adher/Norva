# A Guide-Reading Workflow for Shared Viewing Plans

> **In short:** Select one exact service occurrence, convert its full start and end for everyone, confirm who can join and what “on time” means, then check access separately. Choose a fallback and a person responsible for a final schedule recheck. A shared plan is a dated decision, not a guarantee that the published schedule will remain unchanged.

Group viewing fails when people agree on a title but not the service, event, time zone, start boundary, or acceptable arrival time. A compact decision card resolves those details before they become last-minute messages.

## Fill in the shared-viewing decision card

| Field | Agreement |

| Participants |  |

| Access readiness checked |  |
| Primary device and fallback |  |
| Alternative event or activity |  |
| Recheck owner and deadline |  |

The card needs no sensitive account details. Record only the minimum status required for the plan.

## Choose the exact occurrence

Use the upcoming guide view and select one service, date, and interval. [The current-versus-upcoming decision guide](/blog/compare-current-upcoming-programs/) helps set a useful horizon. If the title repeats, compare event identifier, episode data, and service rather than saying “the evening showing.”

## Convert times for participants

Store one reference timestamp, then convert with named-zone rules for each location. Include dates. The IANA Time Zone Database tracks date-dependent changes, so a fixed offset copied from an earlier month may be wrong.

Ask everyone to confirm the displayed local start and end, not just the start hour. An overnight event can fall on different local dates.

## Agree on attendance boundaries

Define:

- the earliest people can be ready;
- whether joining after start is acceptable;
- the latest acceptable end;
- break or accessibility needs;
- whether the plan proceeds if one person cannot attend.

Do not treat these preferences as guide metadata. They are household decisions attached to a schedule occurrence.

## Verify access separately

A listing does not guarantee playback. Run [the listing-versus-access evidence stack](/blog/guide-listing-vs-media-access/) on the intended account, source, and primary device before the event. Do not share credentials among participants or place them in the decision card.

Norva lets users connect compatible sources they own or are authorized to access. Exact availability remains source- and account-dependent.

## Choose a fallback

The fallback can be:

- another verified occurrence;
- an on-demand item already available through an authorized source;
- a different activity;
- rescheduling the group.

Choose it before the start. A fallback should not require a new search while everyone waits.

## Recheck close to the event

Assign one person to check service, event identity, start, duration, source freshness, and access state at a specified time. If the schedule changed, use [the last-minute revision workflow](/blog/handle-last-minute-schedule-changes/) and communicate the exact difference.

The DVB-I specification distinguishes published and optional actual timing. Keep those labels visible when a late update arrives.

## Communicate one decision message

Use a concise format:

“Program X, Service Y, Saturday 20:00–21:30 Europe/Paris; join by 19:55; final check at 19:30; fallback Z.”

This example shows structure only. Add the real verified event details from the guide.

## Close the plan afterward

Record whether the event started as listed, which fallback was used, and whether any participant’s progress or preferences need correction. Remove temporary reminders that no longer serve a purpose.

## Common mistakes and limitations

- Agreeing on title without service and date.
- Sending one local time to several zones.
- Assuming a listing guarantees access.
- Having no owner for the final check.
- Choosing a fallback after failure.
- Keeping account secrets in a shared note.

## Frequently asked questions

### How early should the final recheck happen?

Choose a time close enough to catch late updates but early enough to communicate and use the fallback.

### What if participants use different devices?

Verify access and output readiness on each required device, or designate one shared primary setup.

### Should the plan include a progress state?

Only if the chosen fallback is an episodic item and household viewing state needs deliberate updates.
