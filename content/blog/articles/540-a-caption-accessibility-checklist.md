# A Caption Accessibility Checklist

> **In short:** Verify that captions are discoverable and correctly identified; include necessary speech, speaker, sound, and music information; remain synchronised and readable; support useful size, contrast, background, and placement; work with intended inputs and responsive layouts; and satisfy real viewer tasks. Record not tested rather than guessing, and separate supplied-resource issues from player presentation.

This checklist turns the [complete caption accessibility guide](/blog/the-complete-guide-to-caption-accessibility/) into a repeatable review.

## Use four statuses

- **confirmed:** task succeeds with direct evidence;
- **issue:** task fails or creates a documented barrier;
- **not tested:** no evidence yet;
- **not applicable:** context truly does not apply.

Never convert not tested into pass.

## Track discovery and identity

- [ ] Caption selector is discoverable with intended inputs.
- [ ] Track language and role are understandable.
- [ ] Selected state is visible and programmatically available where relevant.
- [ ] Captions are distinguishable from translation-only or limited subtitle tracks.
- [ ] Duplicate labels receive controlled cue inspection.
- [ ] State remains clear after menu close.

## Speech, speaker, and sound content

- [ ] Ordinary dialogue is represented in sampled scenes.
- [ ] Off-screen and ambiguous speakers are identifiable when needed.
- [ ] Overlapping or rapid speakers remain understandable.
- [ ] Relevant non-speech sounds are conveyed.
- [ ] Music information is included when meaningful.
- [ ] Cues avoid premature story disclosure.
- [ ] Missing content is reported as a resource finding, not automatically a player defect.

## Timing and reading load

- [ ] Cue onset relates usefully to speech or sound.
- [ ] Cue duration supports reading completion.
- [ ] Beginning, middle, and end samples reveal no unexplained drift.
- [ ] Dense cues do not require repeated rewinds in the tested task.
- [ ] Rapid speaker turns remain attributable.
- [ ] Playback speed is fixed during comparison.

## Typography and line layout

- [ ] Size works at real viewing distance.
- [ ] Font and character shapes remain clear.
- [ ] All-caps blocks receive task-based review.
- [ ] Line length supports scanning.
- [ ] Phrase breaks are understandable.
- [ ] Two- or three-line blocks do not create harmful occlusion.
- [ ] Multilingual scripts, direction, punctuation, and characters render correctly.

## Contrast, edge, and background

- [ ] Bright, dark, detailed, mixed, and moving scenes are sampled.
- [ ] Text remains separable from video.
- [ ] Edge or shadow does not distort characters.
- [ ] Background opacity improves reading without hiding essential visuals.
- [ ] Numeric contrast evidence uses a disclosed method.
- [ ] Viewer task results supplement paused-frame measurement.

## Placement and visual content

- [ ] Captions avoid important faces, signs, lower-thirds, controls, and action.
- [ ] Supported repositioning remains stable.
- [ ] Captions do not jump unnecessarily.
- [ ] Size is not reduced below readability merely to avoid overlap.
- [ ] Different aspect ratios and relevant modes are tested.

## Original evidence: readiness board

| Dimension | Task | Status | Evidence | Owner | Next test |
|---|---|---|---|---|---|
| Discovery | Select and return | Confirmed/issue | Focus path | Person | Action/date |
| Content | Speech/speaker/sound set | Status | Timestamps | Owner | Action |
| Timing | Seven-scene review | Status | Worksheet | Owner | Action |
| Readability | Size/contrast/layout | Status | Scene set | Owner | Action |
| Responsive/device | Small, TV, browser | Status | Task card | Owner | Action |

## Controls and input

- [ ] Pointer, keyboard, touch, and remote are tested where intended.
- [ ] Focus remains visible.
- [ ] Focus returns predictably after the selector closes.
- [ ] Back or Escape behavior is safe.
- [ ] State does not rely on colour alone.
- [ ] Status changes are conveyed appropriately.

## Device and responsive contexts

- [ ] Real small-screen task is tested.
- [ ] Sofa-distance TV task uses the actual remote.
- [ ] Browser resize records meaningful breakpoints.
- [ ] Browser zoom is tested separately from viewport size.
- [ ] Resume, version, device, and eligible offline state are verified only where relevant.

Use [the small-screen caption check](/blog/a-caption-readability-check-for-small-screens/) for mobile evidence.

## User involvement and privacy

- [ ] Caption users define the outcomes that matter.
- [ ] Participation and communication are accessible.
- [ ] Consent and disclosure choices are respected.
- [ ] Individual preferences are not presented as universal.
- [ ] Findings distinguish observation, interpretation, and recommendation.
- [ ] Reports exclude diagnoses, account data, source addresses, and private history.

Use [the user-respecting caption review method](/blog/how-to-run-a-caption-review-without-speaking-for-users/) for research structure.

## Common mistakes and limitations

Avoid dialogue-only testing, one-frame contrast checks, mouse-only controls, desktop-only device simulation, and “users want” claims without evidence.

The caption resource supplies editorial content. Current player and device support determine presentation and controls; this checklist does not manufacture missing cues or certify untested contexts.

## Frequently asked questions

### Must every checklist item pass for every title?

Applicable access requirements should be addressed, but record title-resource findings separately from reusable player controls.

### Can automated testing replace caption users?

No. Automation can support code and contrast checks, while user tasks reveal comprehension, effort, and real strategies.

### How should an unresolved issue be reported?

Provide the smallest reproducible context, task, expected outcome, observed barrier, and redacted evidence without guessing the cause.
