# How to Know When a Catalog Is Clean Enough

> **In short:** A catalogue is clean enough when important household tasks work reliably, broken relationships and destructive risks are resolved, accepted exceptions are documented and owned, and further cosmetic changes deliver less value than their review cost and regression risk. Define that finish line before cleanup, validate it after an observation window, and move remaining low-impact work into routine maintenance.

“Perfect” is not a stable destination. Sources refresh, metadata changes, and household needs evolve. The useful goal is a catalogue that is understandable, retrievable, maintainable, and safe to change.

## Define essential tasks

List what the catalogue must let people do. For example:

- find a known film by title;
- browse a meaningful category;
- distinguish editions or versions;
- reach the correct series and episode;
- identify available language choices;
- resume a partly watched item;
- understand which source or version is selected.

Choose tasks that reflect actual use, not every theoretical path. Assign an owner and a representative test item to each one.

## Create the clean-enough exit scorecard

Rate each dimension **Pass**, **Tolerate**, or **Fail**:

| Dimension | Pass | Tolerate | Fail |
|---|---|---|---|
| Retrieval | Essential tasks work | Rare route has documented workaround | Common task is blocked |
| Relationships | Samples and boundaries resolve | Owned low-impact exception | Orphans or wrong parents affect use |
| Metadata | Key labels are understandable | Minor cosmetic inconsistency | Identity or choice is misleading |
| Personal context | Expected state is retained | Agreed non-critical exception | Progress or favourites are unexpectedly lost |
| Recovery | Baseline and rollback are retained | Recovery is documented but untested for low-risk change | Destructive change lacks recovery |
| Operations | Intake rules and owners are clear | Scheduled follow-up exists | Issue is actively compounding |
| Documentation | Decisions and counts reconcile | Minor note remains | Scope or outcome is unknown |

Any Fail blocks closure. Tolerate requires a reason, owner, and review date; it is not a hidden failure.

## Set an exception budget

An exception budget defines what may remain without undermining the catalogue. Use categories rather than one misleading percentage:

- zero tolerance for wrong-item playback or accidental deletion risk;
- very low tolerance for broken series relationships and missing personal context;
- bounded tolerance for uncertain external metadata;
- higher tolerance for harmless punctuation or artwork variation.

The budget should tighten around consequences, not visual uniformity. Apply [the backlog priority method](/blog/prioritize-catalog-cleanup-backlog/) to anything deferred.

## Measure marginal value

Before opening another batch, ask:

1. Which essential task becomes better?
2. How many verified cases are affected?
3. What new regression risk appears?
4. Can the result be tested and reversed?
5. Is this a one-time correction or a recurring source behaviour?

If the only benefit is tiny cosmetic consistency, while the action touches many stable records, stop. The NDSA Levels of Digital Preservation frame preservation as graduated practices rather than a single all-or-nothing state; catalogue maintenance similarly benefits from explicit, achievable levels.

## Require a quiet observation window

Run [the final quality checklist](/blog/catalog-cleanup-quality-checklist/), refresh connected views normally, and leave the last batch in observation long enough to expose mapping or source-refresh problems. The appropriate duration depends on how often the catalogue changes.

Close only when counts reconcile, no new high-impact exception appears, rollback remains available, and another person can understand the decision record.

## Convert the project into maintenance

Move accepted issues to a dated watchlist, archive completed batch records, and schedule [a monthly cleanup routine](/blog/monthly-catalog-cleanup-routine/). Define triggers that reopen a project: source migration, repeated partial imports, growing orphan counts, widespread metadata mismatch, or a major household workflow change.

Stopping is a controlled handoff, not abandonment.

## Common mistakes and limitations

- Using “zero duplicates” as the only success measure.
- Continuing because tools make more edits possible.
- Hiding unresolved destructive risks as tolerated exceptions.
- Closing immediately after the last change.
- Leaving deferred work without an owner.
- Reopening stable records for stylistic preference alone.

The scorecard reflects local tasks and evidence. It cannot guarantee that an independent source will never change afterwards.

## Frequently asked questions

### Must every metadata field be complete?

No. Prioritise fields needed for identity, retrieval, choice, and accessibility. Missing decorative or unavailable metadata can remain when its effect is understood.

### What if new clutter appears immediately?

Treat it as evidence of an intake rule, mapping, or refresh issue. Diagnose the recurring cause rather than repeating manual cleanup.

### Can “clean enough” differ by household?

Yes. Essential tasks, languages, profiles, and tolerance differ. The finish line should be written, testable, and agreed by the people affected.
