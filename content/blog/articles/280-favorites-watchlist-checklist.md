# A Favorites and Shortlist Quality Checklist

> **In short:** Every saved item needs a future action, exact identity, owner or shared scope, correct list destination, tested retrieval route, and review or expiry trigger. Keep durable favorites, next-up, tonight, and themed shortlists distinct. Block bulk cleanup whenever identity, version, profile scope, or synchronization remains unknown.

This checklist treats curation as a small state system. A visually tidy grid can still fail when it opens wrong versions or mixes several viewers’ preferences.

## Set up the review sheet

| Gate | Status | Evidence | Action |
|---|---|---|---|
| Intent | Pass / fail / unknown |  |  |
| Identity and version |  |  |  |
| Owner and scope |  |  |  |
| Correct destination |  |  |  |
| Retrieval and availability |  |  |  |
| Lifetime and review |  |  |  |
| Exceptions |  |  |  |
| Safe removal |  |  |  |

Date the review and identify the account, confirmed profile, device, and source context.

## Gate 1: intent

- [ ] Each durable favorite has a future-action verb.
- [ ] Temporary curiosity is not described as long-term preference.
- [ ] Completion and favorite state remain separate.
- [ ] Recommendations gain independent intent before saving.

Use [the complete favorites curation guide](/blog/favorites-curation-guide/) for the operating loop.

## Gate 2: identity and version

- [ ] Work title, media type, and release context are distinguishable.
- [ ] Series-level and episode-level saves match their purpose.
- [ ] Remakes and duplicate-looking versions remain separate until compared.
- [ ] Required edit, audio, and subtitles are recorded.
- [ ] A saved card reopens the intended entity.

DCMI metadata terms distinguish identifier, type, relation, format, and language; use those fields when available.

## Gate 3: owner and scope

- [ ] Active viewer or profile is visible before changes.
- [ ] Profile isolation has been tested, not assumed.
- [ ] Shared favorites have agreed ownership and removal rules.
- [ ] Personal fallback notes contain minimal private detail.

## Gate 4: correct destination

- [ ] Favorites represent durable preference or retrieval.
- [ ] Next-up represents a near-term commitment.
- [ ] Tonight’s list expires after the decision.
- [ ] Themed shortlists include a brief and occasion expiry.
- [ ] Investigation items are not presented as ready to watch.

## Gate 5: retrieval and availability

- [ ] Known items can be found within reasonable actions.
- [ ] Filters expose active scope and a clear-all action.
- [ ] Current source availability is separate from favorite identity.
- [ ] Work-level saves do not claim version persistence without evidence.
- [ ] Device migration has a reference snapshot and sample check.

## Gate 6: lifetime and review

- [ ] Every list has a review cadence or event trigger.
- [ ] Temporary candidates expire automatically or by an owner.
- [ ] Quarterly audits use keep, move, merge, investigate, or remove.
- [ ] List size is judged by retrieval capacity, not a universal limit.

Use [the manageable-list test](/blog/keep-favorites-list-manageable/) to measure scan time and unresolved share.

## Gate 7: exceptions

- [ ] Missing favorites are investigated before re-adding.
- [ ] Unavailable items preserve identity.
- [ ] Duplicate-looking items use a work-and-version diff.
- [ ] Source changes are processed in small batches.
- [ ] New-device differences are not assumed to be permanent loss.

## Gate 8: safe removal

- [ ] One non-critical removal proves scope.
- [ ] History and progress are checked separately.
- [ ] Another viewer’s state is protected.
- [ ] Destructive consequences are labelled clearly.
- [ ] A recovery record exists when undo is unavailable.

W3C label guidance supports controls that communicate action and target list. “Remove from tonight” should not imply “delete favorite.”

Norva may retain favorites and other account state across supported devices and organize compatible authorized sources. Exact profile, sync, and custom-list behavior should be verified against current support documentation.

## Decide whether the system passes

Intent, identity, owner, retrieval, and safe-removal scope are critical. A failure or unknown blocks bulk changes. A missing expiry is critical for temporary lists but may be a scheduled-review issue for durable favorites.

Run [the quarterly audit](/blog/quarterly-favorites-audit/) after the checklist identifies a backlog.

## Common mistakes and limitations

- Passing a gate based on appearance.
- Combining durable and temporary intent.
- Ignoring version-specific needs.
- Assuming profiles and sync behavior.
- Removing before preserving identity.
- Measuring quality only by list count.

## Frequently asked questions

### Must every favorite have a written note?

No. Capture only enough intent and identity to support retrieval and review.

### How often should the checklist run?

After source or device changes, before bulk cleanup, and when retrieval quality declines.

### Can software automate every gate?

No. Metadata checks can help, but household ownership, future intent, and ambiguous versions need human decisions.
