# How to Avoid Recency Bias While Browsing a Library

> **In short:** Make older works visible deliberately. Define recent, middle, and older date bands relative to your own library, inspect the same number of candidates from each under one viewing brief, and apply identical relevance and version checks. Do not equate display prominence with quality, or an older date with importance.

This guide uses “recency bias” operationally: a browsing pattern that repeatedly exposes newer records while older ones receive little inspection. It does not diagnose a psychological condition or claim why the interface orders cards.

## Define source-relative date bands

Inspect the earliest and latest verified dates in the relevant part of the connected source. Divide that span into three understandable bands. The bands need not be equal calendar decades; they should each contain enough accurately dated items for comparison.

Mark missing or conflicting dates as Unknown. DCMI defines issued and date-related metadata concepts, while EIDR distinguishes work and version entities whose dates may not mean the same thing. Do not compare an edition date with an original work date without labelling it.

## Hold the discovery brief constant

Use one question across bands, such as “Find a 90-minute-or-shorter work with the required subtitles and a verified subject relation to Seed A.” Keep viewer, media type, language, source, and supported screen fixed.

If older entries lack required metadata, record Metadata Unknown rather than rejecting them as irrelevant.

## Inspect an equal candidate sample

Choose the same small number of titles from each band through verified sorting, filters, search, or category browsing. Apply the same fields:

| Field | Recent | Middle | Older |
|---|---|---|---|
| Relationship evidence |  |  |  |
| Brief fit |  |  |  |
| Version ready |  |  |  |
| Metadata sufficient |  |  |  |
| Shortlist decision |  |  |  |

The point is equal attention, not a quality contest. Do not publish the sample as a general library distribution.

## Change routes when visibility is uneven

If a default browse view remains recent-heavy, use date filters or search to retrieve older works, then follow related titles from a verified older seed. [The search-browse-recommendation selector](/blog/recommendations-search-or-browse/) clarifies each route.

Build [a back-catalog relationship ladder](/blog/discover-back-catalog-related-titles/) to preserve a creator, subject, series, or period link while moving backward in time.

Norva’s exact sort options and date filters require current-product verification. Its source-based discovery remains bounded by the user’s compatible source and metadata.

## Compare outcomes without age stereotypes

Ask whether each candidate answers the brief, has explainable metadata, and is ready in the exact version needed. Do not describe older media as automatically classic, obscure, slower, or superior. Likewise, do not describe recent media as automatically relevant.

Feed unexpected patterns into [the relevance audit](/blog/audit-recommendation-relevance/) and report source/date limits.

## Separate exposure from selection

Record which candidates were visible before any scrolling or route change, then record which ones ultimately passed the brief. A recent title can receive more exposure yet still lose the readiness comparison; an older title can require deliberate retrieval yet become the best fit. Keeping these columns separate prevents the audit from mistaking interface prominence for viewer preference or final relevance.

## Original evidence: date-band card

Complete the matched table with three candidates per band where the source permits. Hide dates from a second reviewer and ask them to classify brief fit and readiness. Restore dates only after the decisions.

The exercise shows whether initial browsing exposure affected this session’s shortlist. It cannot establish personal bias, universal quality, or interface causality.

## Common mistakes and limitations

- Defining old and recent without inspecting source range.
- Mixing work and edition dates.
- Giving newer titles a larger sample.
- Treating unknown dates as old.
- Changing the brief between bands.
- Assuming current interface order reflects importance.

## Frequently asked questions

### Must every date band contribute a finalist?

No. Give each band equal consideration, then keep only candidates that pass the same brief and readiness checks.

### What if the source contains very few older works?

Document the coverage limit. Do not manufacture equal representation or infer missing titles.

### Can release year alone guide discovery?

It can define a range, but combine it with a meaningful relationship and version requirements.
