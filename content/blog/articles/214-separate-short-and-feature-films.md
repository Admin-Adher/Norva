# When to Separate Short Films From Feature Films

> **In short:** Separate shorts and features when people repeatedly browse them for different sessions, work-type metadata is reliable, and the separate view reduces effort without breaking programs or relationships. Do not use an assumed universal runtime threshold. Preserve the work type and runtime as distinct fields, test both combined and separated views, and document how borderline and mixed-program records are handled.

Short and feature can describe work types, distribution contexts, or practical runtime expectations. A personal library needs an operational rule that fits its own records, not an unsupported claim that one duration defines every context.

If most evidence favors separation, create a view or filter before building a completely independent hierarchy.

## Distinguish work type from runtime

EIDR's public FAQ lists referent type and approximate length as separate metadata fields. This supports a useful principle: type and duration are related but not interchangeable.

A short work can be long relative to other shorts; a feature can be brief. Record both when available, and define any local runtime bands as planning aids rather than universal identity rules.

## Test recurring retrieval tasks

Run both views against real questions:

- Find something for a 30-minute session.
- Browse all works by one filmmaker.
- Build a program of several shorts.
- Compare a short and feature on the same theme.
- Find documentary works regardless of length.

If a separate short-film view solves several tasks without harming cross-form discovery, it is likely useful. If runtime filtering already solves the time question, another permanent boundary may be redundant.

Use [the runtime shortlist workflow](/blog/build-runtime-based-movie-shortlist/) for session planning rather than relying only on type labels.

## Preserve programs and components

A compilation, anthology film, or recorded program can contain several shorts. Decide whether the library represents the whole program, its components, or both. Link components with explicit part relationships instead of detaching them silently.

The FIAF cataloguing manual supports hierarchical description and component treatment for moving-image materials. A personal workflow can remain lighter while preserving the parent–part connection.

## Handle borderline and unknown records

Write a local policy:

- which authoritative work-type field is preferred;
- whether runtime is used only as a fallback;
- how unknown type remains discoverable;
- where mixed programs appear;
- how a corrected type is reviewed.

Do not force an unknown record into “feature” simply because that is the default view.

## Choose implementation depth

From lightest to heaviest:

1. Keep one library and use a type filter.
2. Add saved views for shorts and features.
3. Add a short-film project or recurring program list.
4. Build separate subcollections only when ownership and maintenance are clear.

For documentary-heavy holdings, [the documentary subcollection workflow](/blog/organize-documentary-subcollection/) keeps form, subject, and runtime separate.

## Reconnect through projects

Even after separation, a temporary theme may need both forms. [The theme-without-clutter workflow](/blog/track-viewing-themes-without-clutter/) creates a cross-form manifest without weakening the durable type boundary.

Norva may provide type, runtime, filtering, and version grouping over compatible authorized sources, but available values depend on source metadata. Test known shorts, features, and mixed programs before trusting a separated view.

## Common mistakes and limitations

- Treating one runtime threshold as universal.
- Using duration as the only evidence of work type.
- Breaking a program into unlinked items.
- Hiding unknowns in the default feature view.
- Creating separate hierarchies before testing saved views.
- Losing cross-form creator or theme discovery.

## Frequently asked questions

### What runtime defines a short film?

Definitions vary by context. Use an authoritative work-type source and document any local runtime rule explicitly.

### Should a collection of shorts be one feature record?

Preserve the program as a work when appropriate and link identifiable component works rather than flattening the relationship.

### Can shorts and features still appear together?

Yes. Cross-form creator, subject, and project views should remain possible even when default browsing separates them.
