# Record a Progress Baseline After a Source Change

> **In short:** Select a tiny authorized sample and record the profile, privacy-safe source label, item or episode, visible version, duration, saved position or completion state, device, application version, and timestamp before the source change. Afterward, locate the same logical item and verify its source and version identity before comparing progress. Repeat one sample on another supported device if relevant. Record reset, preserved, duplicated, moved, or unknown without exporting the household's full history.

Progress is associated with an item identity and profile, not just a title a person recognizes. A source change can expose different versions or identifiers for apparently similar media entries.

## Choose a minimal sample

Select one partly watched item, one completed item, and one episodic example when those cases matter. Use sample codes instead of sensitive titles. Do not manufacture extensive viewing history for testing.

The [catalog-baseline guide](/blog/baseline-before-second-source/) captures broader category and filter state.

## Confirm the profile

Record the active profile and verify it after the change. A different profile can have distinct history, progress, favorites, and preferences. Do not merge profile evidence into an account-wide assumption.

Norva's current privacy notice describes progress as usage data that can synchronize and personalize the experience.

## Record visible item identity

Capture title metadata, year, season and episode where relevant, duration, version label, source label, and any ordinary UI distinction. Do not inspect hidden source databases or extract private identifiers.

A similar title and artwork may still represent another edition or episode.

## Record position carefully

Use the displayed time, percentage, completion mark, or progress bar available. Note the format and timestamp. Do not convert a rough visual bar into false numerical precision.

If the source change occurs during playback, stop and save state first.

## Change one source variable

Follow the [small-sample verification guide](/blog/verify-sample-after-source-connection/) and keep profile, filters, grouping, device, and application version stable. Add, update, or remove only the planned source configuration.

Do not clear history or mark items completed during the comparison.

## Match identity before progress

After the change, confirm the sample belongs to the expected source and version. If another source supplies the same title, compare duration, metadata, episode, and version before deciding progress reset.

The future [item identity change guide](/blog/progress-reset-after-item-identity-change/) provides a recovery audit for a genuine mismatch.

## Compare supported devices

If cross-device continuity matters, check one other supported trusted device with the same profile and item. Record timestamps and application versions. Do not promise a synchronization delay that current official guidance does not state.

A one-device difference can be local stale state rather than source identity.

## Classify the result

Use preserved, reset, duplicated, moved to another version, profile mismatch, device-specific, or unknown. Preserve a redacted screenshot only if needed for support.

Never publish complete history or source credentials.

Keep every sampled item's before and after observations together as an identity pair. Include the profile, privacy-safe source label, visible title or sample code, episode when relevant, version cues, duration, saved position, completion state, device, application version, and timestamp. This makes a changed position interpretable: the apparent reset may belong to another version or episode rather than the original entry. Do not copy an entire watch history or expose unrelated household viewing habits. When identity remains uncertain, classify continuity as unknown and attach only the minimal pair to a focused support request.

## Common mistakes and limitations

- Comparing different profiles.
- Matching only by title or artwork.
- Recording false precision from a progress bar.
- Changing filters and history during the test.
- Exporting complete household viewing history.
- Assuming one-device state is synchronized truth.
- Calling a version mismatch permanent data loss.

## Frequently asked questions

### Why can the same title show different progress?

Different sources, versions, episodes, durations, or item identities can produce separate progress associations.

### Should I inspect hidden identifiers?

No. Use ordinary UI and official support evidence; do not extract private source data or bypass application controls.

### How much progress data should I record?

Use a tiny representative sample with masked titles and only the fields needed for comparison.
