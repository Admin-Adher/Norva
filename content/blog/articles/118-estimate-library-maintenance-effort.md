# How to Estimate the Work Required to Maintain a Library

> **In short:** Separate fixed recurring reviews from per-item work. Time a representative batch containing normal items and exceptions, estimate how many arrive in a typical cycle, add coordination and follow-up, and report a range rather than one precise number. If the high end exceeds available household time, simplify the rules or reduce scope.

Maintenance effort is created by decisions, not only collection size. A small multilingual library with inconsistent sources can require more review than a larger stable collection with clean metadata.

## List fixed and variable work

**Fixed work** occurs even when few items arrive:

- source status review;
- storage or recovery check;
- seasonal meeting;
- decision log review;
- retrieval test;
- archive review.

**Variable work** grows with arrivals or changes:

- intake classification;
- metadata correction;
- duplicate comparison;
- language verification;
- exception handling;
- category placement;
- migration reconciliation.

Estimate these groups separately.

## Sample real work

Choose a batch containing:

- several normal items;
- missing metadata;
- a possible duplicate;
- a multi-version title;
- a multilingual item;
- an unavailable item;
- one source change.

Time each action category, not just total session length. Record interruptions separately. Do not claim that the sample represents every future cycle; keep low, typical, and high observations.

## Estimate a range

Use:

`cycle effort = fixed reviews + routine items + exceptions + coordination + follow-up`

For each variable part:

`estimated range = expected count × observed low-to-high time`

Do not add decimal precision that the sample cannot support. Round to a practical planning range and note the sample date, sources, and rules used.

## Include exception and coordination cost

An item that enters a review queue still consumes future time. Track:

- evidence requested;
- person who can decide;
- approval delay;
- retest;
- documentation;
- rollback preparation.

Shared collections also need communication. The [responsibility matrix](/blog/assign-shared-library-responsibilities/) makes these handoffs visible.

## Compare effort with capacity

Record the household's realistic maintenance capacity per cycle. Do not assume all free time is available. Keep a buffer for unusual source changes or recovery tasks.

If projected effort exceeds capacity:

- reduce mandatory metadata;
- process smaller batches;
- archive inactive groups logically;
- lengthen low-risk review intervals;
- improve source-level rules;
- assign a second trained maintainer;
- narrow the current scope.

Use [the priority framework](/blog/choose-collection-planning-priorities/) to protect high-impact work when capacity is limited.

## Calibrate repeated work separately

The first attempt at a task often includes learning, documentation, and tool setup that later batches will not repeat. Keep three timings:

- first-time setup and rule clarification;
- routine execution after the rule is understood;
- rework caused by an exception or incorrect decision.

Do not remove setup time from the plan; assign it to the project phase where it occurs. Do not assume routine work will keep accelerating indefinitely either. Fatigue, difficult source data, household approval, and quality checks place practical limits on batching.

After two cycles, compare forecast and actual effort by task type. If the estimate was wrong, update the arrival count, exception rate, or task definition rather than adding an unexplained percentage. Preserve the old estimate and reason for revision so later maintainers understand the change.

## Measure value alongside effort

Lower effort is not the only goal. A rule may be worth maintaining if it prevents frequent wrong-version choices or keeps household profiles distinct. Compare time with [library success measures](/blog/choose-library-success-measures/).

Remove rules that consume recurring effort without supporting an essential retrieval, safety, or resilience outcome.

## Common mistakes and limitations

- Multiplying one easy item's time across the whole collection.
- Ignoring exceptions and approvals.
- Reporting a precise total from a tiny sample.
- Treating volunteer time as unlimited.
- Forgetting fixed seasonal and source checks.
- Optimising speed while accuracy declines.
- Keeping mandatory fields no one uses.

The estimate describes the current rules and sources. Recalculate after migration, scope, source, or household changes.

## Frequently asked questions

### How large should the sample be?

Large enough to include routine items and every common exception type. Stop when additional items no longer reveal new work categories, while acknowledging uncertainty.

### Should waiting time count?

Record it separately. It affects elapsed completion but not always active effort. Both matter for planning a review queue.

### What if the estimate is too high?

Simplify metadata, narrow scope, reduce low-risk frequency, or distribute documented work. Do not hide the gap by leaving exceptions unowned.
