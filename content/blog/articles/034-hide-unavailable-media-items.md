# Why Hiding Unavailable Items Makes a Library Easier to Browse

> **In short:** Hide unavailable entries when the immediate goal is to choose something usable now. Keep them visible when you need catalogue context, want to diagnose a source issue, or expect availability to return. Hiding should be a reversible view preference, not a substitute for understanding why an item is unavailable.

An unavailable card still consumes attention. In a long row, several such entries can make a collection feel larger without increasing the number of viable choices. A focused view can reduce that friction, but a complete view answers different questions.

## Separate visibility from identity

Three actions are often confused:

- **Hide:** temporarily remove an entry from the current browsing result.
- **Remove from a category:** change an organisational relationship.
- **Delete at the source:** change the underlying catalogue through the source's authorised controls.

These actions have different consequences. A browsing filter should not imply that an item has been deleted or that its metadata no longer exists.

Norva organises the catalogue supplied by a compatible source that the user is authorised to use. The source controls the underlying media access.

## Choose the view that matches the task

Use a playable-only view when:

- people are choosing something for immediate playback;
- unavailable entries dominate the first rows;
- the same unavailable state has already been understood;
- remote-control browsing needs to stay concise.

Use the complete view when:

- diagnosing why an item cannot be used;
- checking whether a source connection changed;
- comparing multiple versions;
- maintaining catalogue structure;
- looking for a title whose availability may have changed.

The guide to [why library items can be unavailable](/blog/why-library-items-are-unavailable/) covers diagnosis rather than presentation.

## Run the two-view availability protocol

Select one representative category and keep its sorting unchanged.

### View A: complete catalogue

Record:

- how many unavailable entries appear before a usable option;
- whether unavailable states are clearly marked;
- whether grouped versions behave differently;
- whether the list provides useful historical context.

### View B: playable-only

Enable the available-item view if the current interface offers it, then record:

- whether choosing becomes faster or simply less informative;
- whether expected titles disappear;
- whether current filters remain understandable;
- whether the control can be reversed easily.

Compare the observations. The protocol produces a decision for that category and task; it does not claim that hiding is always superior.

## Diagnose unexpected gaps before reorganising

If a title you expected is absent, clear the availability control first. Then:

1. confirm that the compatible source is still connected;
2. inspect whether the title or version is marked unavailable;
3. check other active filters;
4. compare another supported device if relevant;
5. use the source's own authorised controls for source-level issues.

The [source-connection verification guide](/blog/verify-media-source-connection/) offers a structured check. Do not create duplicate entries or change categories until the availability state is understood.

## Account for grouped versions

One version may be unavailable while another remains usable. A grouped title should make that distinction visible instead of hiding the entire work because one option cannot be selected.

Open the group, identify the available variant, and confirm that language or other required attributes still match. If the grouping makes this impossible to understand, revisit the group rather than turning off availability filtering globally.

## Keep personal views distinct

Recent, Favorites, and All Titles answer personal browsing questions; availability answers whether the current source can support the next action. These controls can work together, but each should remain understandable.

For example, an unavailable favourite may still belong in Favorites as a reminder, while a playable-only filter removes it from tonight's candidate list. Compare these modes in [Recent, Favorites, or All Titles](/blog/recent-favorites-all-titles/).

## Common mistakes and limitations

- Assuming a hidden entry was deleted.
- Leaving the filter active and later believing the catalogue is incomplete.
- Reorganising categories to solve a temporary source state.
- Treating all versions as unavailable because one version is.
- Hiding entries before collecting enough information to diagnose them.
- Expecting availability to be identical across every source or media item.

## Frequently asked questions

### Is hiding unavailable media the same as removing it?

No. Hiding changes the current browsing view. Removing or deleting changes organisation or the underlying source and should be treated as a separate action.

### Why might an item become available again?

Availability depends on the connected source and its conditions. A connection, source state, rights, or version can change. Confirm the current source information rather than assuming permanence.

### Should unavailable favourites stay visible?

They can remain useful as personal context. Apply an availability view when choosing something now, and return to the complete Favorites view when maintaining the list.
