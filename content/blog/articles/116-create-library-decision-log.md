# Why a Library Decision Log Prevents Repeated Rework

> **In short:** A decision log preserves the reason behind category, metadata, source, archive, and migration choices. Record the problem, options considered, evidence, selected rule, affected scope, owner, approval, rollback route, and review trigger. Future maintainers can then apply or revise the rule without reopening the same debate from memory.

Collections change slowly enough that people forget why a choice was made, yet quickly enough that the choice returns. A short log turns institutional memory into household documentation without creating a complex management system.

## Log decisions, not every action

Record a decision when it:

- changes the collection scope;
- adds or removes an authorised source;
- defines a shared category;
- maps metadata values;
- merges or separates versions;
- moves a group between active and archive;
- approves a migration;
- permits a destructive cleanup;
- resolves a recurring exception;
- changes household responsibility.

Routine actions that simply follow an existing rule do not need separate entries. Link the entry to the rule or batch instead.

## Capture the context

Every entry should answer:

1. What problem or trigger caused the decision?
2. Which items, sources, profiles, or categories are affected?
3. What evidence was available?
4. Which options were considered?
5. Why was the selected option preferred?
6. Who executed, approved, and reviewed it?
7. How can the change be reversed?
8. When should it be reviewed?

Do not put credentials, tokens, private source addresses, or personal viewing details beyond what the decision genuinely needs.

## Distinguish facts, assumptions, and preferences

Label evidence as:

- **Observed:** reproduced on a named device or sample.
- **Source fact:** verified through current official documentation.
- **Household preference:** agreed way of browsing or labelling.
- **Assumption:** plausible but not yet verified.
- **Unknown:** evidence not available.

This distinction prevents a preference from later being treated as a technical requirement. It also makes assumptions easy to revisit.

## Link decisions to governing documents

The [shared collection charter](/blog/create-media-collection-charter/) should hold stable operating rules. The decision log explains changes and exceptions. The [metadata rulebook](/blog/plan-metadata-rules-before-import/) defines field-level behaviour. Avoid copying the same rule into all three places; link them and identify which document is authoritative.

## Make entries easy to retrieve

Use a stable identifier such as `DEC-YYYY-001`, a date, short title, topic label, and status. Suggested status values:

- proposed;
- approved;
- implemented;
- superseded;
- reversed;
- review due.

Do not delete superseded entries. Link the old and new decisions so the history remains understandable.

## Original evidence: decision log template

```text
Decision ID / date / title:
Trigger and problem:
Affected scope:
Evidence (observed / official / preference / assumption / unknown):
Options considered:
Decision and reason:
Executor / approver / reviewer:
Rollback or reversal route:
Related charter, rule, or migration batch:
Status / review trigger / review date:
```

Create one entry for a recent repeated disagreement. Ask another household maintainer to locate it and explain the rule without verbal help. If they cannot, improve labels or storage location.

## Use the log during review

At each [seasonal library review](/blog/plan-seasonal-library-reviews/):

- close implemented entries;
- identify assumptions still unverified;
- review decisions whose trigger occurred;
- link superseding entries;
- remove no history unless an appropriate privacy or retention decision requires it;
- compare the decision with current retrieval results.

Use [library success measures](/blog/choose-library-success-measures/) to decide whether the change worked, not whether the entry was completed.

During review, sample one reversed decision and confirm that a reader can follow the original evidence, superseding entry, effective date, and current rule without relying on household memory.

## Common mistakes and limitations

- Logging every minor action until the system becomes unusable.
- Recording a decision without its reason.
- Mixing credentials with source documentation.
- Deleting superseded history.
- Calling assumptions facts.
- Making a reviewer infer affected scope.
- Keeping the log where only one person can find it.

A decision log records household reasoning; it does not create legal rights, verify source authorisation, or guarantee technical compatibility.

## Frequently asked questions

### How long should an entry be?

Usually a few structured paragraphs or one form. Add detail only when it supports future review, rollback, or accountability.

### Should rejected options be recorded?

Yes, briefly. Knowing why an option was rejected prevents the same proposal from returning without new evidence.

### Can a decision be reversed?

Yes. Create a new linked entry stating the trigger, evidence, reversal action, and resulting rule. Preserve the earlier entry as history.
