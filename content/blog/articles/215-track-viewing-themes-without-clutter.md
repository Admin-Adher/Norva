# How to Track a Viewing Theme Without Creating Category Clutter

> **In short:** Treat a viewing theme as a dated project, not permanent metadata. Write the question, membership rule, source scope, owner, candidate roles, review date, and exit criteria. Add a movie only when its role is explicit, track viewing state inside the manifest, then close and archive the project. Promote a theme to durable metadata only after it supports recurring retrieval beyond one project.

Themes are intentionally interpretive: “stormy coastlines,” “unreliable narrators,” or “workplace transformation” may be useful for a month and irrelevant afterward. Permanent categories should not have to carry every temporary question.

The theme question should be answerable through viewing. A label without a question invites unlimited accumulation.

## Give every candidate a role

Use a role ledger:

| Movie | Why it belongs | Evidence | Role | Version ready | Viewing state |
|---|---|---|---|---|---|
|  |  |  | Anchor / contrast / edge case / conclusion |  |  |

Remove a candidate when its role duplicates another and adds no useful contrast. The goal is a coherent project, not exhaustive tagging.

For a period theme, [the decade retrospective grid](/blog/build-decade-movie-retrospective/) adds a coverage audit. For a subject-heavy documentary project, [the documentary subcollection workflow](/blog/organize-documentary-subcollection/) helps distinguish durable subject metadata from the current theme.

## Keep durable metadata separate

Stable facts—title, creator, release date, runtime, language, version relationship—belong on the work or version. Interpretive project facts—why selected, expected contrast, sequence, current question—belong in the manifest.

Dublin Core separates properties such as subject, temporal coverage, relation, and description. A personal workflow can similarly avoid turning one project note into a universal subject claim.

## Use dated review points

At the review date, ask:

- Does the question still matter?
- Does every candidate have a distinct role?
- Are version and availability checks current?
- Is the target size realistic?
- What has been watched, skipped, or added?
- Is the project moving toward its exit criterion?

Do not let a dormant theme remain an active category simply because closing it feels final.

## Track states explicitly

Use a small set: candidate, verified, scheduled, watched, skipped, removed. State changes should have a reason when they affect the argument.

W3C notification guidance recommends clear outcomes and recovery actions. Apply the same principle: “removed from theme, remains in library” is clearer than making a poster disappear without explanation.

## Define promotion criteria

Promote an element of the theme into durable library structure only if:

- it answers recurring retrieval questions;
- its meaning can be defined consistently;
- source coverage is adequate;
- multiple future projects will use it;
- the maintenance owner and correction rule are clear.

Usually the project itself should remain archived while a proven subject, relationship, or creator field survives.

## Close and archive

When the exit criterion is met, record completion date, watched sequence, key conclusion, unresolved candidates, and any durable metadata corrections. Freeze membership so the archived project remains intelligible.

Before archiving, reopen one completed and one rejected candidate to confirm that their recorded roles still explain the theme's conclusion.

For a franchise project, [the completed film-series archive workflow](/blog/archive-completed-film-series/) preserves routes and branch decisions in a more relationship-focused form.

Norva may organize favorites, versions, and compatible authorized sources across supported devices, but temporary project behavior should be verified against current features. A simple external manifest remains valid when a dedicated saved collection is unavailable.

## Common mistakes and limitations

- Creating a category before defining the question.
- Adding candidates without a distinct role.
- Mixing interpretive notes with stable factual metadata.
- Leaving projects active with no review date.
- Promoting a one-time theme into permanent taxonomy.
- Deleting the project instead of archiving its conclusion.

## Frequently asked questions

### How long should a temporary theme last?

Set a review date and exit criterion based on the project. Duration alone does not determine value.

### Can one movie belong to several themes?

Yes. Each manifest should state a separate role; that does not require multiple permanent categories.

### What if the theme becomes useful repeatedly?

Promote only the reusable, well-defined field or relationship and keep individual project manifests archived.
