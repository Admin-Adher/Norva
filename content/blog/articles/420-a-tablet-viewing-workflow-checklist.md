# A Tablet Viewing Workflow Checklist

> **In short:** Before viewing, verify profile, item, version, posture, power, network or local state, audio, and subtitles. During interruptions, recheck identity and position before resuming. At close, confirm progress, remove temporary local items, restore changed settings, and sign out when the tablet is not trusted. Mark unknowns instead of guessing.

This checklist is the operational companion to the [complete tablet viewing workflow](/blog/the-complete-guide-to-tablet-viewing-workflows/). Use it for a real session, not as a promise that every tablet or source behaves identically.

## Before opening Norva

- [ ] The tablet is a supported device and shows no unresolved system warning.
- [ ] The verified app or official web route is available.
- [ ] The device lock and privacy state suit the environment.
- [ ] Power is sufficient or the charging cable is routed safely.
- [ ] The viewing position is stable and essential controls are reachable.
- [ ] The intended network is available, or eligible local media has been tested.
- [ ] Essential notifications remain allowed.

If the session will be long, run the fuller [long-session tablet preparation](/blog/how-to-prepare-a-tablet-for-a-long-viewing-session/).

## Account and library entry

- [ ] The intended account is active.
- [ ] The intended profile is visibly selected.
- [ ] The connected media source is one you own or are authorised to use.
- [ ] The library section matches the task.
- [ ] Active filters and sort state are understood.
- [ ] A scroll landmark is noted before opening a detail screen.

Norva is a software media player and organiser. The subscription covers software features and does not include a media catalogue.

## Item and version preflight

- [ ] Full title is confirmed.
- [ ] Year is checked for films when available.
- [ ] Series, season, and episode are checked for episodic media.
- [ ] The grouped version or source label is confirmed.
- [ ] Approximate starting position is understood.
- [ ] The selected audio track is available.
- [ ] The selected subtitles are available or intentionally off.
- [ ] Any visible quality or availability label is treated as information, not a guarantee.

Languages and subtitles depend on the source and media. Offline access depends on the device, source, and associated rights.

## Playback start

- [ ] Sound is routed to the intended speakers or headphones.
- [ ] Starting level is low and comfortable.
- [ ] A short test shows the expected item and position.
- [ ] Pause and return controls are reachable.
- [ ] The timeline responds to one deliberate play/pause cycle.
- [ ] No unexpected account, source, or security message appears.

If a row fails, stop and change one state at a time. Repeated tapping is not a diagnostic method.

## During an interruption

For screen lock, rotation, app switching, network change, or audio-output change:

- [ ] Pause if a reliable control is available.
- [ ] Record the current item and approximate position.
- [ ] Complete the interruption.
- [ ] Reconfirm profile.
- [ ] Reconfirm title, episode, and version.
- [ ] Compare playback position.
- [ ] Recheck audio route, audio track, and subtitles.
- [ ] Resume once and wait for a visible response.

W3C orientation and focus guidance describe accessible behaviour, but exact recovery depends on the device and interface.

## Before a handoff

- [ ] The target is a supported device.
- [ ] The same authorised source is accessible.
- [ ] The intended account and profile are known.
- [ ] Item, episode, version, and approximate position are recorded.
- [ ] Audio and subtitle needs are recorded.
- [ ] The source device is paused before the target begins.
- [ ] Concurrent-use conditions are not inferred from profile count.

A handoff should be treated as a new verification, not as an automatic extension of the source screen.

## Session close

- [ ] Playback is paused.
- [ ] Final position is visible or noted.
- [ ] Temporary favourites or planning notes are resolved.
- [ ] Eligible local items are removed when appropriate.
- [ ] Notification, lock, or orientation changes are restored.
- [ ] The profile is handed off deliberately.
- [ ] The account is signed out on an untrusted or borrowed tablet.
- [ ] Reopening does not expose a private session when sign-out was intended.

## If a check keeps failing

Write one symptom, preserve the first evidence, and use the [tablet playback problem report](/blog/how-to-document-a-tablet-specific-playback-problem/). Include environment, exact steps, expected result, observed result, and one controlled comparison. Remove all credentials and private data before sharing.

## Common mistakes and limitations

Do not treat a checked box as proof beyond the tested session. Updates, network changes, different media, and another profile can change the result. Avoid checking items from memory, skipping identity after an interruption, or using a checklist to bypass a warning.

This checklist does not certify hardware, accessories, media rights, or source compatibility.

## Frequently asked questions

### Must every line pass before a short browse?

No. Mark irrelevant lines not applicable. Identity, account, privacy, and any action that could change progress still deserve attention.

### Should I keep completed checklists?

Keep only what helps future sessions and contains no sensitive data. A short baseline may help troubleshooting; delete private notes when no longer needed.

### Is the checklist a substitute for support?

No. It helps isolate state. Persistent errors, account concerns, or unclear warnings should be reported through the verified support path.
