# How to Evaluate Norva for a Multilingual Household

> **In short:** Inventory the audio and subtitle tracks that the authorized source actually provides before setting preferences. Define each household language scenario, choose representative media and versions, then test selection, persistence where supported, subtitle readability, screen switching, and missing-track fallback. A preference cannot create a track, so source coverage is a non-negotiable input to the decision.

Multilingual viewing can mean original audio with translated subtitles, dubbed audio without subtitles, captions in the spoken language, or different choices for different profiles. A single "language supported" score hides these distinct needs.

Use the [Norva evaluation framework](/blog/norva-evaluation-framework/) to make essential language and accessibility scenarios stop conditions.

## Map household scenarios

For each recurring viewing context, record preferred audio, preferred subtitles or captions, whether text is essential, primary screen, and acceptable fallback. Use role-based profile labels rather than sensitive identity details.

Do not assume that all household members prefer the interface language to match media audio or subtitles. Test those layers separately.

## Inventory source tracks

Through the source's authorized workflow, choose five representative items and record the actual audio and subtitle labels for each version. Include a movie, a series episode, a variant case, an item with the preferred combination, and an item missing one desired track.

Norva does not supply the media catalog or language tracks. Availability depends on the source, media, version, rights, device, and current supported behavior.

## Check labels and identity

Compare the source labels with what Norva presents. Determine whether abbreviations are understandable to the household. Confirm the selected media version before concluding that a track is missing.

Similar editions can carry different dubs, captions, subtitle formats, or timing. Record version identity in every language test.

## Test audio preferences

On one intended profile, set the preferred audio language and test a known compatible item. Exit and reopen it, then test a second item containing that track. Record automatic selection, manual selection, fallback, or unavailable.

Use the [first audio preference guide](/blog/set-first-audio-preference/) to keep profile, source, version, and screen conditions explicit.

## Test subtitle preferences and readability

Set the subtitle preference separately. Watch three short dialogue intervals and record visibility, broad timing, line clipping, and readability without transcribing the content. Repeat after reopening the item.

The [first subtitle preference guide](/blog/set-first-subtitle-preference/) includes a reusable readability record. For caption-dependent viewing, treat timing and visibility as access requirements rather than optional quality points.

## Test profile separation

If household members use different language choices, establish one known item and preference under each relevant profile. Switch profiles and verify that the intended context remains understandable. Do not assume profiles are separate security accounts.

Check the current official plan for profile limits. Choose a plan because tested contexts need it, not merely because more profiles are available.

## Test multiple screens

Repeat one essential language scenario on the primary and secondary supported screens. Inspect the active track, actual audio, subtitle visibility, and any preference behavior. Keep account, profile, source item, and version fixed.

Device and format capabilities can affect results. A successful mobile test does not prove identical TV or browser behavior.

## Test missing-track fallback

Use the selected item known not to contain the preferred track. Record the interface's actual behavior: another available language, subtitles off, a prompt, or an unknown state. Decide whether that fallback is understandable and acceptable for the intended viewer.

Do not label the case a synchronization or preference failure merely because the source lacks the track.

## Review navigation and shared-screen clarity

Ask each participant to select their profile, find the known item, inspect the version, and reach the preferred tracks using the normal remote, touch, or browser input. Count actions and errors. Long labels, hidden controls, or unclear focus can make an available track practically inaccessible.

Review privacy before sharing screenshots that reveal household languages, profiles, or viewing choices.

## Frequently asked questions

### Can Norva add a missing dub or subtitle track?

No. Preferences select among source-provided, supported tracks; they cannot create audio or text absent from the media.

### Does one language label guarantee the same track across versions?

No. Editions can differ. Verify the exact media version, format, and actual playback choice.

### Should each language receive its own profile?

Only when a separate recurring context reduces conflicts and fits current profile limits. A preference can belong to a profile without making language the profile's only purpose.
