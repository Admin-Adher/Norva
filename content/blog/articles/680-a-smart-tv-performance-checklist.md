# A Smart TV Performance Checklist

> **In short:** Define one symptom and observable endpoint, then record TV and app versions, lifecycle, remote, focus, rendering, artwork, search, storage warnings, network path, authorised media version, playback controls, and output. Run matched controls before escalating through reversible recovery. Preserve timing ranges, failures, privacy boundaries, and unknowns rather than guessing a hidden cause.

This checklist is a routing tool. Complete only the sections relevant to the symptom, and stop when a boundary is clear enough for focused guidance or support.

## 1. Freeze the initial state

- Record TV model, system version, app version, trusted installation source, time and zone.
- Label cold launch, warm launch, resume, or screen revisit.
- Note storage warnings, active network category, output route, remote type, and recent changes.
- Write the exact expected and actual behavior in one sentence.

Use the [layer-by-layer TV guide](/blog/smart-tv-media-app-performance-a-layer-by-layer-guide/) when several sections appear relevant.

## 2. Define one measurable workflow

Choose a visible start and end: keypress to settled focus, screen appearance to defined artwork, final search character to stable results, play to first frame, or command to media-state change. Run a small fixed count and preserve every result.

W3C Performance Timeline provides timing concepts for supported web implementations; manual television timing remains approximate.

## Original evidence: ordered worksheet

| Layer | Fixed observation | Matched control | Result | Confidence/limit |
|---|---|---|---|---|
| Remote/focus | One key to target | TV settings | Range | Manual timing |
| Rendering/artwork | Defined tiles settle | Cached revisit | Range/failures | Hidden stages |
| Search/network | Character and results | Local action/path | Range | Endpoint differs |
| Media/control | Version and timecode | Another version | Result | Media mismatch |
| Output | Picture/audio route | Supported local route | Result | Handshake hidden |

Attach the completed worksheet privately only when official support requests it.

## 3. Check remote and focus

Use isolated directional presses from named elements. Record ignored, queued, repeated, asymmetric, or incorrect movement. Compare ordinary TV settings with the same remote. Do not flood input or use a service menu.

If focus is correct but late only while tiles load, note the shared screen state.

## 4. Separate artwork and search

For artwork, define visible tiles and distinguish blank, temporary, final, and failed states. For search, separate input echo, query delay, text results, artwork, and layout. A fast speed test cannot prove the relevant resource or search endpoint responded quickly.

Keep the query privacy-safe and crop viewing history from evidence.

## 5. Review storage and app state

Record only official free-space values and warnings, update activity, repeated reloads, and state loss. Do not infer memory pressure from storage or set an arbitrary free-space threshold. Avoid cleaners, data clearing, reinstall, and factory reset during the initial pass.

Use the [monthly maintenance routine](/blog/a-monthly-smart-tv-app-maintenance-routine/) for ordinary upkeep rather than diagnostic resets.

## 6. Compare the network boundary

Record wired or Wi-Fi category, broad household load, source status, and time. Compare a primarily local action with artwork, search, source refresh, or playback. Change one supported path component only when it will not disrupt another service.

Repeat in alternating order and retain latency variation or failure, not throughput alone.

## 7. Match media and output

Record authorised title version, duration, verified codec and profile when relevant, resolution, frame rate, dynamic range, audio, subtitles, timecode, and output. W3C Media Capabilities describes contextual capability questions for supported environments, not universal proof.

Keep volume safe and do not disconnect active equipment without manufacturer guidance.

## 8. Escalate recovery carefully

Retry once from the known state, restart only the app after evidence, then use the manufacturer's supported TV restart if needed. Retest the exact workflow after each action. Stop before data clearing or reinstall unless consequences and recovery are documented.

Record the outcome as an association: “did not recur after restart in three matched trials,” not “restart proved the cause.”

## 9. Package a private report

Summarize symptom, first occurrence, environment, workflow, raw ranges, failures, controls, impact, accessibility impact, recovery order, and unknowns. [A performance diary](/blog/how-to-keep-a-smart-tv-performance-diary/) can provide the change timeline.

RFC 6973 supports minimizing disclosure. Remove account IDs, source URLs, network names, addresses, serials, notifications, and unrelated history.

Before publication, verify any current Norva feature, compatibility, telemetry, and support path against official product documentation.

## Frequently asked questions

### Must every checklist section be completed?

No. Complete the minimum sections needed to isolate the observed boundary.

### Does one successful control identify the cause?

No. It narrows scope while differences between the control and failing case remain.

### When should troubleshooting stop?

Stop when recovery risks data, accessibility, household services, or unsupported changes, and use official support.
