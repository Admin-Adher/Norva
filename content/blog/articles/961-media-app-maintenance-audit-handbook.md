# The Complete Media App Maintenance and Audit Handbook

> **In short:** Use a light monthly account-and-device review, a quarterly profile audit, and event-driven checks after application updates, system updates, router changes, source changes, password changes, or household changes. Review storage and language preferences on a cadence tied to actual use. Capture a small baseline before changing anything, test one layer at a time, record only non-sensitive evidence, and assign every unresolved result.

Media applications sit between accounts, devices, networks, sources, and household habits. Maintenance is not a single “clear everything” action. It is a small set of proportionate checks that preserve a known-good baseline and reduce risky improvisation when something changes.

## Establish four separate layers

Keep account, application, device, and connected-source observations in separate columns. Add network and household administration when relevant. A playback issue after a router change should not be labeled an account failure without evidence, and missing source metadata should not be treated as a device-storage problem.

Norva organizes and plays media from compatible sources users own or are authorized to use; it does not supply a media catalog. That boundary belongs in every maintenance record.

## Use a monthly light review

Once a month, confirm the expected account owner, supported devices currently in use, sign-in or session information visible through official controls, subscription-provider awareness, and any unexplained device relationship. Do not invent a device-management screen if one is not documented.

The [monthly account and device review](/blog/monthly-account-device-review/) provides a minimal evidence sheet.

## Run a quarterly profile audit

Every quarter, verify that each profile has a recurring purpose, privacy-safe name, responsible administrator, and expected preferences. Review obsolete or duplicate contexts without copying viewing history. Norva includes up to two profiles and Norva Family up to five; capacity is not a device or simultaneous-playback claim.

Use the [quarterly profile hygiene audit](/blog/quarterly-profile-hygiene-audit/) for a controlled decision.

## Trigger checks after updates

After an application update, run a short smoke check covering launch, account context, one catalog route, one known playback sample, track selection where relevant, clean exit, and return. After a device system update, add permissions, output, storage, network, and device-specific behavior.

The [post-update smoke check](/blog/post-app-update-smoke-check/) prevents a full reset from replacing evidence.

## Recheck after network changes

A router replacement or network reconfiguration can alter addressing, name resolution, filtering, isolation, and source reachability. Record the old and new broad context, test the source independently, then test the application. Never place wireless passwords or private addresses in the audit.

Use a separate network-change checklist instead of changing source credentials preemptively.

## Audit after source changes

Before adding, removing, refreshing, or changing a source, capture a small catalog, category, identity, track, progress, and favorite baseline. Afterward, compare known samples rather than only total counts.

The [catalog-wide source-change audit](/blog/catalog-audit-after-source-change/) links identity, hierarchy, metadata, and user state without assuming every source value stays fixed.

## Escalate after credential events

After an account password change or suspected exposure, review supported device and session information available through official controls, recovery settings, shared screens, and source-secret separation. Do not assume a password change automatically ends every session.

The [trusted-device audit](/blog/trusted-device-audit-after-password-change/) provides the event-driven sequence.

## Maintain storage deliberately

Review storage when free space crosses a household threshold, before travel, after legitimate offline use, or on a recurring cadence supported by actual accumulation. Offline access remains conditional on the device, source, and associated rights. Do not delete unfamiliar application data without understanding sign-in and offline consequences.

Mobile and TV storage need separate evidence because device controls and constraints differ.

## Audit audio and subtitle preferences

Use known media with confirmed source tracks. On each required supported screen, check availability, selection, and preference behavior where those options exist. A preference cannot create a missing language or subtitle track.

The [subtitle and accessibility preference audit](/blog/subtitle-accessibility-preference-audit/) treats access needs as explicit pass conditions.

## Retain less evidence

Keep dates, broad device categories, application versions, neutral sample codes, observed results, and owners. Exclude credentials, tokens, private source addresses, full catalogs, detailed viewing histories, and unnecessary household identities. Delete temporary screenshots after the support or security need ends.

## Common mistakes and limitations

- Running destructive cleanup before capturing a baseline.
- Mixing source, network, device, and account changes.
- Storing secrets in maintenance evidence.
- Auditing every possible screen instead of required workflows.
- Treating profile capacity as a device limit.
- Keeping screenshots indefinitely without a purpose.

## Frequently asked questions

### Must every check run monthly?

No. Keep the monthly review light and use quarterly or event-driven audits for deeper or less frequent risks.

### Should maintenance include a full reset?

Not routinely. A reset is disruptive and may remove useful state; isolate the affected layer and follow current official support guidance first.

### Can one audit prove every device is healthy?

No. Test representative required screens and record the scope. Expand only when a device represents a distinct essential workflow.
