# How to Review Outdated Version Groups Safely

> **In short:** Do not retire a version because its poster, label, or source looks old. Inventory every member of the group, confirm that they represent the same work, compare edition, language, subtitle, accessibility, technical, source-health, and personal-context differences, then classify each as preferred, useful alternative, duplicate candidate, broken, or unresolved. Change only a small reversible pilot.

Version grouping reduces repeated cards while keeping meaningful alternatives available. An apparently redundant entry may be the only one with a preferred language, subtitle track, edit, or reliable source route.

## Confirm the group identity first

Compare title, release date, creators, runtime range, edition wording, series placement, and stable source identifiers. A remake, extended edition, special, or similarly named episode should not be treated as a duplicate merely because artwork matches.

If identity is uncertain, remove the item from the cleanup batch and investigate. Metadata disagreement is evidence, not permission to choose the neatest-looking card.

Use language labels carefully. The W3C explains that language tags identify human languages and may include script or region subtags; a broad “English” label and a specific tagged variant are not automatically interchangeable.

Norva can group variants from compatible authorised sources and retain language or subtitle preferences, but the tracks actually available depend on the source and media.

## Classify value rather than age

Assign one evidence-backed status:

- **Preferred:** the version selected for normal use.
- **Useful alternative:** retains a unique language, edition, accessibility feature, quality, or source route.
- **Duplicate candidate:** same work and edition with no verified meaningful difference.
- **Broken candidate:** inaccessible or malformed, with the cause recorded.
- **Unresolved:** conflicting identity or incomplete evidence.

Age can be a clue about source health, but it is not a classification. A newer record may be incomplete, while an older one may preserve the only suitable track.

## Check personal and household context

For every preferred or duplicate candidate, inspect progress, history, favourite state, and profile-specific choice. Use [the personal-context map](/blog/cleanup-without-losing-personal-state/) before changing record identity.

If the chosen preferred version changes, test whether the household can still find and resume the intended item. Do not assume the group card transfers state between independent source records.

## Choose the least disruptive action

Possible outcomes include correcting labels, regrouping a misplaced version, keeping a useful alternative, hiding a broken entry while it is investigated, or retiring a proven duplicate reference. Follow [the hide-remove-delete decision guide](/blog/hide-remove-delete-differences/) for scope.

When forming a new group, use [the multiple-version grouping guide](/blog/group-multiple-media-versions/) and begin with one group. Preserve the evidence sheet and rollback route until the observation window closes.

## Validate the pilot

After the supported refresh:

1. search for the work;
2. confirm only the intended group is shown;
3. open each retained version;
4. verify labels match actual selectable tracks;
5. check progress and favourites for relevant profiles;
6. repeat on a second supported view;
7. restore the baseline if any expected option disappears.

Record exceptions separately. A version that fails playback requires diagnosis; it does not prove the other version is equivalent.

## Common mistakes and limitations

- Selecting the highest resolution without checking language or edition.
- Comparing posters instead of identifiers.
- Removing a record that holds personal state.
- Treating runtime difference as definitive proof by itself.
- Grouping a film with a remake or similarly named episode.
- Testing only the preferred version after cleanup.

Metadata may be incomplete or incorrect. The worksheet supports a careful decision but cannot establish identity when authoritative evidence is absent.

## Frequently asked questions

### How old must a version be before review?

Age is not the trigger. Review when a source has changed, the group is confusing, playback fails, labels conflict, or the catalogue has a documented cleanup cycle.

### Should a lower-quality version always be removed?

No. It may be more compatible, carry a needed track, or serve as a reliable fallback. Compare actual household requirements.

### What if two versions appear identical?

Check stable references, edition, runtime, tracks, playback, and personal context. If no meaningful difference is verified, mark one as a duplicate candidate and pilot the least destructive supported action.
