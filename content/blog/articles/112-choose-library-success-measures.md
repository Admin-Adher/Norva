# How to Measure Whether a Library Plan Is Working

> **In short:** Repeat the same essential retrieval tasks before and after a pilot. Measure correct item or version found, errors, assistance, unresolved queues, maintenance time, source coverage, and whether changes remain reversible. A plan is working when important household tasks improve without shifting the burden into hidden cleanup or risky maintenance.

Counting renamed items rewards activity, not usefulness. Measurement should return to the purpose of the collection: helping people find, choose, resume, and maintain authorised media with enough context.

## Start with a stable test set

Choose representative items before organisation begins:

- one exact title;
- one episode;
- one version group;
- one multilingual item;
- one unavailable item;
- one favourite or unfinished item under a profile;
- one archive candidate;
- one known metadata gap.

Record source, version, profile, device, and starting screen. Reuse the same test set after the change. Otherwise, a different result may reflect different inputs rather than the plan.

## Measure retrieval outcomes

For each task, record:

- correct item found: yes or no;
- correct version identified before playback;
- number of wrong turns;
- assistance required;
- major screen changes;
- confidence reason, such as source or language evidence;
- recovery after an error.

Do not impose one universal speed target. Remote, touch, and keyboard tasks differ. Define household thresholds in [the findability goals guide](/blog/set-library-findability-goals/).

## Measure exception health

A clean-looking library can hide unresolved work. Track:

- uncategorised items;
- possible duplicates;
- missing metadata;
- uncertain rights or source ownership;
- failed imports;
- stale decisions;
- archive items with no review date.

The goal is not always zero. A clearly owned review queue is healthier than incorrect forced decisions. Measure age, owner, and next action.

## Measure maintenance cost

Record actual minutes spent on:

- intake;
- metadata correction;
- category review;
- duplicate review;
- source checks;
- archive review;
- household coordination.

Also record how many decisions were completed and how many new exceptions appeared. The [maintenance effort guide](/blog/estimate-library-maintenance-effort/) helps distinguish one-time migration work from recurring cost.

## Measure coverage and resilience

Ask whether:

- every in-scope source has an owner;
- every critical source has a current inventory entry;
- essential metadata fields are present or explicitly unknown;
- important changes are logged;
- rollback or export exists before broad edits;
- source change triggers are documented;
- another authorised household member can understand the plan.

NDSA preservation guidance treats inventory, storage, integrity, security, metadata, and formats as connected areas. A household does not need institutional metrics, but it should avoid measuring appearance alone.

## Interpret changes carefully

An improvement after one pilot is evidence for that sample, not every future item. Look for:

- repeated result across several viewers;
- improvement without extra assistance;
- exceptions that remain visible;
- maintenance within the agreed budget;
- no loss of source or version context;
- stable results after new items arrive.

If a metric improves only because difficult items were hidden, the plan has not solved the original problem.

## Common mistakes and limitations

- Counting activity instead of outcomes.
- Changing the test set after organisation.
- Averaging essential failures with optional passes.
- Ignoring exception age.
- Measuring only the maintainer's experience.
- Claiming a precise improvement from a small informal sample.
- Treating an empty category or queue as proof of correctness.

Household testing is useful operational evidence, not a formal usability study or universal product benchmark.

## Frequently asked questions

### How many tasks should be measured?

Use three to six essential tasks plus representative edge cases. More tasks are useful only when someone can maintain the measurement.

### Is time-to-find enough?

No. Correct version, independence, errors, confidence, and maintenance cost can be equally important.

### What if viewers disagree about success?

Keep results by person or profile. Investigate the workflow difference rather than averaging away a legitimate need.
