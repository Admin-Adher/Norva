# TV Guide Time Zone Wrong After Travel? A Recovery Check

> **In short:** Record the device's current clock, named time zone, UTC offset, automatic or manual state, selected guide date and window, source listing times, Norva listing times, device, application version, and travel sequence using only broad zone labels. Verify operating-system time through official controls, reopen the same guide context, and compare another supported device. Do not expose precise locations or manually rewrite source schedule times.

Travel can change a device's clock and time-zone context, but a guide difference after travel does not prove which layer failed to update. A recovery check compares before, destination, and current observations.

## Minimize location data

Use time-zone names or broad labels such as origin zone and destination zone. Do not record hotel, home, workplace, public address, precise coordinates, or unnecessary itinerary details in support evidence.

Norva's privacy policy should guide data minimization. The [TV guide handbook](/blog/tv-guide-troubleshooting-handbook/) provides the complete time matrix.

## Record the travel sequence

Capture last correct guide observation, departure zone, destination zone, when the device clock changed, first incorrect guide observation, return or onward travel, and current state. Add timestamps and observation gaps without claiming continuous monitoring.

## Check operating-system time state

Through the device's official settings, record local date and time, automatic or manual clock, automatic or manual zone, named zone, and UTC offset where visible. Do not disable automatic settings merely to make one guide entry align.

## Verify the guide window

Record selected date, visible guide header, current-time marker where shown, and focused program. Navigate to the intended time using documented controls. A retained prior date or hour is a view difference, not necessarily a zone failure.

## Compare source listing times

Through the provider's official authorized route, record a sampled channel's listing title, start and end times, and any explicitly labeled zone or offset. Do not infer the zone of unlabeled times or edit the source schedule.

## Compare one-hour and larger shifts

If the difference is exactly one hour, use the [one-hour offset checklist](/blog/guide-listings-shifted-one-hour/) to include daylight-saving context. If it matches the travel-zone difference, record that correlation without declaring conversion behavior.

## Reopen one stable context

After verifying device time settings, return to the same account, profile, source, channel group, filters, date, and guide window. Record the result. Do not change profile, source, filters, clock, and application state simultaneously.

## Compare another supported device

Use the same account, profile, source, channel, and close timestamp. Record each device's clock, named zone, offset, operating system, and application version. The [cross-screen guide check](/blog/guide-works-on-one-device-only/) helps preserve equivalent contexts.

## Preserve application transitions

Record whether the application remained open during travel, entered the background, was closed, resumed after device sleep, or was updated. These are context changes, not proof of an internal cause.

## Avoid broad resets

Do not change source times, channel identifiers, device location permissions, network security, application data, or account settings as first steps. Avoid repeated refreshes before the original sequence is saved.

## Classify recovery

Use device clock incorrect, named zone incorrect, offset differs, guide window retained, source time unlabeled, one-device difference, corrected after official device-time update, corrected after documented navigation, not reproduced, or unknown. Do not promise automatic zone behavior not documented by Norva.

## Prepare support evidence

Use the [guide evidence record](/blog/guide-issue-support-evidence/) with broad zone labels, clock and offset, guide window, sampled source and Norva times, devices, versions, travel sequence, and actions. Remove exact locations and credentials.

## Common mistakes and limitations

- Sharing precise travel or home locations.
- Changing source schedule times to compensate.
- Comparing different guide dates or filters.
- Turning off automatic time without preserving baseline.
- Assuming a correlation proves conversion behavior.
- Resetting application data before cross-screen comparison.

## Frequently asked questions

### Should support receive my exact location?

No. A named time zone, offset, device context, and timestamps are usually the relevant evidence; minimize location detail.

### Should I set the clock manually?

Follow the device manufacturer's official guidance. Record the current state before changing it and avoid manual compensation as a shortcut.

### What if only one device has the wrong guide time?

Compare clocks, zones, versions, guide windows, profiles, and close timestamps before concluding which layer differs.
