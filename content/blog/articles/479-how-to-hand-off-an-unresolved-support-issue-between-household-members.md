# How to Hand Off an Unresolved Support Issue Between Household Members

> **In short:** Transfer a single redacted packet containing the exact symptom, expected result, environment, timeline, evidence, changes already tried, current baseline, support reference, and safest next action. Name the new coordinator, confirm consent from affected owners, and prohibit duplicate or destructive changes until the incoming person has read back the handoff.

A handoff should preserve reasoning, not just screenshots. “It still does not work” forces the next person to repeat tests and may erase the state support needs to understand.

## Pause before transferring ownership

The outgoing coordinator should stop new experiments and record the current state. Tell household members who now owns the incident and when the transfer becomes effective.

If the issue involves another person's profile, source, device, or billing context, obtain permission before sharing evidence or authorising a change. Coordination authority is not unlimited account authority.

## Build the handoff packet

Include seven parts:

1. **Symptom:** exact visible or audible behavior.
2. **Expected result:** what should have happened.
3. **Environment:** device, operating system, app or browser version, account context, anonymised profile, source, item, and media version.
4. **Timeline:** last success, first failure, and local time zone.
5. **Evidence:** exact error text and privacy-safe screenshots.
6. **Attempts:** one line per change, result, and rollback.
7. **Current baseline:** the state the incoming owner will inherit.

Use the structure in [documenting a profile-specific problem](/blog/how-to-document-a-profile-specific-problem-for-support/) when the issue follows one profile.

## Separate facts from hypotheses

Mark observations as facts: “Track label appeared, then playback used another language.” Mark explanations as hypotheses: “Selection may not have persisted.” Do not present a guessed cause as confirmed.

List each hypothesis already tested and the result. If a change was not rolled back, say so clearly.

## Redact before sharing

Remove passwords, payment data, source addresses, account email, private title history, unrelated profile names, and device identifiers that support does not need. Preserve the wording of an error while hiding surrounding private data.

Use the least revealing channel available to the authorised people. Do not create a broad household chat archive for a one-person profile issue.

## Original evidence: read-back handoff

The incoming coordinator should read back:

- the exact unresolved symptom;
- the current device and version context;
- the last change and whether it was reversed;
- the next approved action;
- the rollback route;
- the stop condition;
- the person whose consent is needed.

The outgoing coordinator corrects only factual differences. This read-back exposes missing context before another variable changes.

## Transfer the support conversation

Record the official case reference, last support instruction, action completed, and result. Do not copy a support agent's message out of context. The incoming person should use the same case where the channel permits, identify the ownership change, and avoid opening parallel reports unless instructed.

Follow the household's [one-change troubleshooting rule](/blog/how-to-set-a-household-rule-for-troubleshooting-changes/). A handoff is not permission to reset, reinstall, remove a source, delete a profile, erase a device, or change credentials.

## Close or return the issue

Define a completion test before the next action. If it passes, record the verified context and remove obsolete evidence. If it fails, update the same timeline. If identity, rights, privacy, or rollback becomes uncertain, stop and return to the relevant owner or official support.

Place this workflow within the [shared household governance guide](/blog/the-complete-guide-to-shared-household-media-governance/) so incident roles are known before a problem occurs.

## Common mistakes and limitations

Avoid screenshot-only handoffs, undocumented resets, parallel cases, copied credentials, missing time zones, and claims that a temporary improvement proves the root cause.

A strong packet improves continuity but cannot guarantee resolution. Source behavior, media metadata, device state, and current product controls may all matter.

## Frequently asked questions

### Should the new coordinator repeat all previous tests?

No. Review the recorded evidence first. Repeat only a safe test when its context or result is uncertain and the affected owner agrees.

### Can the packet include a password for convenience?

No. Keep credentials in their official owner-controlled process and transfer responsibility without copying secrets.

### When is a handoff complete?

When the incoming person has accepted ownership, read back the baseline, understands the next approved action, and knows the stop condition.
