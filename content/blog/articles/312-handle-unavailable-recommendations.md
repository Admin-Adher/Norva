# How to Handle a Recommended Title That Is Unavailable

> **In short:** Preserve the title’s identity and why it was recommended, then diagnose availability separately. Clear filters, confirm the connected source and profile, search for the exact work, and compare any authorised variants. Keep an unavailable but relevant title in an investigation list with a review trigger; never present it as ready to watch or assume another version is equivalent.

Unavailability answers “can I use this version now?” It does not answer “was this a relevant discovery lead?” Keeping those questions separate prevents a temporary source state from erasing a useful path.

## Capture the recommendation before changing anything

Record the seed, candidate title, media type, release context, visible relation, source label, version clues, profile, filters, and local time. A cropped image may help, but exclude credentials and unrelated history.

DCMI terms distinguish identifier, relation, format, and language. EIDR’s work hierarchy distinguishes works, edits, and manifestations. These fields let you preserve identity even when the playable record is not currently available.

## Run the availability check

1. Clear search and filters.
2. Confirm the intended profile and compatible source.
3. Search the exact title and an alternate title only when verified.
4. Inspect work, episode, and version matches.
5. Check required language and subtitles.
6. Test the normal detail route without repeated playback attempts.
7. Record observed error or unavailable state exactly.

Do not guess whether the cause is rights, connectivity, source indexing, or product state. Describe only what the interface and source show.

## Compare alternate versions cautiously

Another card may represent the same work but a different cut, episode mapping, duration, language, subtitle set, or source record. Apply [the version-readiness gate](/blog/verify-version-before-following-recommendation/) before treating it as a replacement.

A version is acceptable only when it preserves the viewing purpose and meets the intended viewer’s requirements. “Same poster” and “same title” are insufficient.

## Choose one of four outcomes

| Outcome | Condition | Action |
|---|---|---|
| Ready | Exact suitable version is available | Add to focused shortlist |
| Investigate | Identity or source state is unclear | Preserve evidence; contact support if needed |
| Retain for review | Relevant work is unavailable now | Keep minimal identity and review trigger |
| Release | Relevance or future intent has ended | Remove from temporary discovery list |

If metadata is the uncertainty, use [the incomplete-metadata diagnostic](/blog/recommendations-with-incomplete-metadata/). Do not repeatedly re-add or open the same unavailable card.

## Keep the shortlist honest

Use [the recommendation-to-shortlist workflow](/blog/turn-recommendations-into-shortlist/) to separate Ready from Research. A tonight list should contain only candidates that can actually support the planned session. An unavailable discovery lead can remain valuable in a research ledger without competing with actionable choices.

Norva organises compatible sources and does not include a media catalogue. Availability depends on the user’s source, media, device, and associated rights. Exact unavailable-state behavior requires current product verification.

## Original evidence: two-axis decision tree

Create one axis for Relevance—confirmed, weak, or unknown—and another for Availability—ready, unsuitable version, unavailable, or unknown. Place the affected title and any variants in the grid with evidence.

Ask another reviewer to choose the same outcome using the grid alone. The tree demonstrates a disciplined decision; it cannot prove why availability changed or when it will return.

Date every Retain for Review outcome and name the event that justifies another check, such as a verified source refresh or device change. A recurring calendar check without new evidence only recreates the same unavailable observation.

## Common mistakes and limitations

- Treating unavailability as irrelevance.
- Clearing filters without recording the baseline.
- Accepting another version by title alone.
- Promising that availability will return.
- Keeping unavailable research in a ready-to-watch shortlist.
- Sharing source credentials in support evidence.

## Frequently asked questions

### Should I favorite an unavailable recommendation?

Only when a durable retrieval purpose remains and current product behavior supports it. Otherwise keep a minimal research note.

### Can I search another connected source?

Only sources you own or are authorised to use, and only through verified current controls. Compare exact versions before substituting.

### When should I contact support?

When the exact authorised item is expected to be available, filters and context are verified, and a reproducible unavailable state persists.
