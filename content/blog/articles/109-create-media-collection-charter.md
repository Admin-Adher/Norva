# Create a Simple Charter for a Shared Media Collection

> **In short:** Put the household's shared rules on one page: purpose, scope, authorised sources, personal versus shared state, decision roles, changes requiring approval, maintenance cadence, rollback expectations, and review date. Keep it practical. A household charter is an operating agreement, not a legal document or a substitute for source terms.

A charter reduces repeated arguments because it preserves the reason behind a rule. It also protects personal viewing state from being treated as shared catalogue maintenance.

## State the purpose and scope

Begin with two sentences:

- **Purpose:** what daily viewing tasks the collection should support.
- **Scope:** which media types, authorised sources, people, and time period the current rules cover.

Link the scope to [the personal media collection plan](/blog/plan-personal-media-collection/). If an item sits outside it, the charter should say “review later,” not force a decision.

## Define shared and personal state

Shared decisions may include:

- source labels;
- category meanings;
- version-grouping rules;
- archive boundaries;
- minimum metadata;
- maintenance schedule.

Personal state may include:

- playback progress;
- history;
- favourites;
- language and subtitle preferences;
- personal discovery choices.

Norva profiles can separate personal state. Norva includes up to two profiles and Norva Family up to five. These are profile capacities, not parental controls, device limits, or simultaneous-playback claims.

Use [the household needs map](/blog/map-household-viewing-needs/) to decide which state needs separation.

## Assign decision roles

Use four simple roles:

- **Maintainer:** performs routine reversible work.
- **Approver:** decides shared or potentially destructive changes.
- **Source owner:** verifies authorisation and access for a source.
- **Reviewer:** checks whether the change improved agreed tasks.

One person may hold several roles, but every action should have a named owner. The [shared responsibility guide](/blog/assign-shared-library-responsibilities/) turns the roles into a task matrix.

## Mark changes that require approval

The charter should require explicit approval before:

- removing a source;
- deleting or permanently altering shared material;
- merging uncertain duplicates;
- changing a category used by several people;
- clearing another profile's state;
- changing archive rules;
- starting a migration;
- sharing source access details.

Routine reversible actions can have standing permission, such as placing uncertain items in a review queue.

## Add safety and privacy boundaries

State that:

- only compatible sources the household owns or is legally authorised to use belong in the plan;
- credentials, tokens, and recovery codes are never stored in the charter;
- evidence screenshots remove private details;
- source and platform terms still apply;
- personal history and preferences are changed only by the appropriate person or authorised maintainer;
- uncertain rights or ownership block migration and deletion decisions.

Do not turn the charter into unsupported legal advice. It records household practice.

## Specify maintenance and review

Include:

- intake cadence;
- review queue owner;
- seasonal review month;
- event triggers;
- maximum maintenance time;
- decision log location;
- charter version and next review date.

Use [the library decision log](/blog/create-library-decision-log/) for exceptions and changes so the charter remains short.

## Original evidence: one-page charter template

```text
Purpose:
Scope:
Authorised sources and owners:
Shared catalogue rules:
Personal profile boundaries:
Routine maintainer:
Changes requiring approval:
Rollback/export requirement:
Review queue location:
Maintenance cadence:
Decision log location:
Version / approved by / review date:
```

Ask every regular viewer to explain the clauses in their own words. If two interpretations differ, rewrite the clause. Pilot it for one maintenance cycle and record exceptions.

Before approval, rehearse one disputed bulk deletion: the charter should identify the decision owner, required evidence, rollback path, and person who confirms recovery.

## Common mistakes and limitations

- Writing abstract values instead of operational rules.
- Calling everything shared.
- Naming “the household” as owner of every task.
- Including credentials or private source details.
- Giving one maintainer permission for irreversible changes.
- Treating profile capacity as device or control capability.
- Never revisiting the charter after sources or people change.

The charter cannot guarantee source availability, metadata quality, or compatibility. It only makes household decisions explicit.

## Frequently asked questions

### Does a small household need a charter?

A short version can still help when more than one person shares categories, sources, or maintenance. It may be unnecessary for a single-person collection with no shared decisions.

### How long should it be?

One page is usually enough. Put detailed exceptions and history in the decision log.

### Who approves the first version?

Every regular viewer affected by the shared rules should review it. Record agreement and unresolved objections without inventing formal legal authority.
