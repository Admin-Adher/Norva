# Where to Find Official Norva Help

> **In short:** Begin at norva.tv and navigate to the current support page from the official site. Describe one problem with reproducible steps, the affected surface, time, and sanitized error. Never send a password, access token, live pairing code, private source address, or complete account identifier. Keep the support confirmation and review what evidence was shared.

Search results, social posts, direct messages, and community replies may look helpful but do not establish an official support relationship. A safe support process verifies the route before discussing the problem. The [first privacy review](/blog/first-norva-privacy-review/) helps identify what should be removed from evidence.

## Start at the official domain

Type or use a trusted bookmark for norva.tv, then follow the current support navigation. Check the domain carefully before signing in or submitting information. Do not trust a shortened link, sponsored result, QR code, or unsolicited message merely because it uses the Norva name or logo.

The official support path may evolve. Always confirm it from the current site rather than copying an old address from a forum post.

## Classify the issue boundary

Decide whether the observed problem concerns the Norva account, profile, application surface, device, network, connected source, source metadata, media format, or rights. You do not need to identify the final cause. The category only helps produce a focused report.

Remember that Norva is software for organizing and playing media from a compatible source the user owns or is legally authorized to use. Norva does not include a media catalog, so source availability and source credentials remain separate concerns.

## Create a minimum reproducible case

Write the smallest sequence that shows the problem:

1. Starting screen and active profile shorthand.
2. Action taken with the intended input method.
3. Expected result based on the visible interface.
4. Actual result or sanitized error.
5. Whether the issue repeats under unchanged conditions.

Record the date, local time zone, broad device category, Norva surface, network type, and item type where relevant. Avoid a long narrative containing unrelated experiments.

## Redact evidence before sending

Review every screenshot, recording, and log. Remove passwords, tokens, pairing codes, source addresses, usernames, account identifiers, payment details, faces, notifications, precise location clues, and unnecessary catalog titles. Crop to the relevant interface area when possible.

Never assume an expired secret is safe to publish. A screenshot can expose patterns, account context, or other still-valid information.

## Preserve state before destructive changes

If the problem involves a missing favorite, wrong progress, device record, or catalog structure, capture the sanitized state before deleting, reconnecting, renaming, or resetting anything. Destructive changes can erase evidence and create a second problem.

The [first-week setup audit](/blog/first-week-norva-setup-audit/) provides a systematic record of which onboarding checks passed and which need support.

## Verify the response channel

Compare any reply with the original official case confirmation. Be cautious if a message asks you to move to an unrelated platform, install remote-control software, disclose a password, purchase an unusual payment method, or share a live code.

Return to the official support page independently if the request seems unexpected. Do not use contact details supplied only inside the suspicious message.

## Close the case responsibly

When the issue is resolved, record the verified cause or action, affected version or environment, and the final test result. Delete redundant raw recordings and retain only the minimum case record needed for future reference.

Use the [Norva onboarding journey](/blog/norva-onboarding-complete-journey/) to resume at the last verified checkpoint rather than restarting every setup decision.

## Common support mistakes

- Following an unsolicited support link.
- Sending passwords, tokens, codes, or source details.
- Reporting many unrelated symptoms in one case.
- Resetting the account before preserving evidence.
- Leaving private notifications visible in a recording.
- Trusting a reply that cannot be tied to the official case.

## Frequently asked questions

### Can official support ask for diagnostic information?

It may request relevant context through the current official process. Provide only the minimum necessary information and never disclose passwords, live codes, or source secrets.

### Is a social-media reply official support?

Do not assume so. Verify the support route independently from norva.tv and continue sensitive discussion only through the confirmed channel.

### What if I cannot determine whether the source or Norva is responsible?

Report the boundary as unknown and provide a source baseline, fixed conditions, reproducible steps, and sanitized result. Good evidence does not require guessing the cause.
