# An Audio Track Management Checklist

> **In short:** Confirm the source, item, version, profile, device, and playback context; read every visible track label; verify language and role with a short sample; choose for the current viewer; recheck after resume, episode, version, device, shared-profile, or offline changes; and preserve exact evidence before reporting. Never guess a role or use destructive troubleshooting first.

This checklist converts the [complete audio-track management guide](/blog/the-complete-guide-to-managing-audio-tracks/) into a repeatable session and diagnostic routine.

## Mark each item clearly

Use four states:

- **confirmed:** directly observed in this context;
- **unverified:** possible but not tested;
- **needs action:** a defined problem or decision exists;
- **not applicable:** the context does not apply.

Do not use “absent” for an entry you have not checked.

## Before opening playback

- [ ] Connected source and authorisation are current according to the owner.
- [ ] Correct title, season, episode, and media version are selected.
- [ ] Active account or profile is appropriate for the viewer.
- [ ] Device and output route are known.
- [ ] Online or eligible offline state is recorded.
- [ ] The viewer's language or accessibility outcome is understood.

Norva is software for organising and playing compatible connected sources; the selected media determines which tracks exist.

## Read the selector

- [ ] Complete visible audio list is inspected.
- [ ] Exact language text is preserved.
- [ ] Role labels such as commentary or audio description are read separately.
- [ ] Channels, codec, or custom title are noted only when shown.
- [ ] Selected state is identifiable without relying on memory.
- [ ] Same-language entries are not assumed equivalent.

Use “unknown role” when metadata does not answer the question.

## Verify the intended track

- [ ] A dialogue-rich or visually informative scene is selected.
- [ ] Spoken language is confirmed.
- [ ] Commentary, audio description, or standard role is verified when relevant.
- [ ] Current output plays dialogue as expected.
- [ ] Subtitles complement the intended audio when used.
- [ ] No subjective quality claim is presented as a technical fact.

## Choose for the session

- [ ] The selection supports comprehension and access needs.
- [ ] Original or dubbed identity is verified before comparison.
- [ ] Shared viewers agree on a workable choice.
- [ ] A title-specific exception is not mistaken for a universal account rule.
- [ ] A shared-profile change is communicated.
- [ ] The end-state or next-viewer recheck is agreed.

## Check persistence boundaries

- [ ] Same title/version reopens with a recorded result.
- [ ] Resume route is checked separately when relevant.
- [ ] Next episode's complete track list is compared.
- [ ] Another version is treated as distinct media.
- [ ] Another device is verified independently of progress.
- [ ] Shared-profile use is considered without inspecting private history.

When a preference appears lost, use the scope matrix rather than changing several settings.

## Original evidence: readiness record

Create one row per meaningful context:

| Context | Target label | Target available | Selected | Heard result | Status | Next action |
|---|---|---|---|---|---|---|
| Item/version/device | Exact text | Yes/no/unclear | Exact text | Verified role | Confirmed/issue | One action |

The record should make unknowns visible and keep credentials, source addresses, account email, and unrelated history out.

## Series and version checks

- [ ] First, middle, and final episodes are sampled when screening a season.
- [ ] Outlier neighbors are checked.
- [ ] Every episode is checked when an access-critical need requires certainty.
- [ ] Each version has its own row.
- [ ] Availability, metadata, selection, and output exceptions remain separate.

Use [the series audio audit matrix](/blog/build-an-audio-track-audit-matrix-for-a-series/) for a repeatable season review.

## Offline readiness

- [ ] Current plan, device, source, and item eligibility are verified.
- [ ] Exact local item and version are confirmed.
- [ ] Local selector is compared with the connected selector.
- [ ] Intended track is sampled without connectivity.
- [ ] Device custody and privacy are appropriate.
- [ ] Evidence is preserved before recreating a local copy.

The [offline audio readiness guide](/blog/how-offline-availability-can-affect-audio-track-choices/) provides the paired connected/local card.

## Troubleshooting and reporting

- [ ] Baseline is captured before correction.
- [ ] One variable changes at a time.
- [ ] Exact steps, expected result, and observed result are recorded.
- [ ] Screenshots are authorised and redacted.
- [ ] No media, credentials, private addresses, or unrelated history are attached.
- [ ] Source removal, data clearing, reinstall, reset, and credential changes require an official rationale and owner authority.
- [ ] Report states the smallest reproducible boundary.

## Common mistakes and limitations

Avoid reading role from order, treating language as origin, assuming season or version consistency, and promising preference persistence across every device.

The checklist verifies one tested context. Current Norva documentation, source metadata, media availability, plan terms, and supported devices remain the authoritative boundaries.

## Frequently asked questions

### Must every check be repeated for every title?

Use the full routine for new, ambiguous, accessibility-critical, or problematic contexts; routine sessions may need only identity, selection, and sample checks.

### What is the most important troubleshooting rule?

Preserve the baseline and change one variable. That keeps results interpretable and protects support evidence.

### Where should an unresolved mismatch go?

Route source-supplied media questions to the source owner and player-context evidence through Norva's official support path, without guessing the cause.
