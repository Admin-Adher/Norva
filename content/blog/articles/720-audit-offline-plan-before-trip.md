# Audit an Offline Viewing Plan Before a Trip

> **In short:** Audit the plan before the final night by challenging every dependency: realistic viewing time, exact item eligibility, completed local copies, required tracks, free storage, battery, profile ownership, fallback, reconnection, and cleanup. Mark each area green, amber, or red from current evidence. Any essential item that remains amber or red must be corrected or removed from the plan.

An audit is different from a checklist. A checklist asks whether a task was performed; an audit asks whether the evidence is strong enough to rely on the result. Run it early enough to change the batch calmly.

## Audit the schedule

Compare planned duration with dependable viewing windows, not total travel time. Remove periods occupied by transfers, sleep, work, meals, or situations where device use is inappropriate.

Ask whether every item serves a named window. Extra items should have a distinct fallback role. The [responsible travel checklist](/blog/offline-viewing-travel-checklist/) provides the operational sequence after this planning review.

## Audit eligibility and identity

For each essential item, verify:

- exact title, episode, and version;
- intended account and profile;
- current offline control;
- completed local state;
- permitted use under the compatible source and associated rights;
- current disconnected test.

Norva offline access depends on the device, compatible source, media, and associated rights. Use the [item-level availability proof](/blog/verify-offline-availability-before-leaving/) rather than a catalogue screenshot.

## Audit storage and device health

Open the operating system's storage view. Compare current free space with the batch's measured local sizes and your protected reserve. Do not rely on one universal size-per-hour assumption.

Check whether pending app or operating-system updates could consume space or change behaviour. Complete necessary updates early enough to retest; avoid unnecessary last-minute changes.

## Audit power and accessories

Review the [offline battery budget](/blog/offline-viewing-battery-budget/) from a representative local playback observation. Confirm the device starts charged, essential-use reserve is protected, and any permitted cable, adapter, headphones, or power bank has been tested.

Mark the power plan amber if it depends on an unverified outlet. Mark it red if viewing would consume battery reserved for essential travel functions.

## Audit audio, subtitles, and profiles

Essential audio and subtitle options must work on the exact local item. Confirm readability, timing, and audio output under expected conditions. On a shared device, identify who owns each item and who may remove it.

Do not assume saved preferences prove local track inclusion or that profiles isolate local media from other device users.

## Audit fallback and reconnection

The plan should survive one unavailable item without risky troubleshooting. Use one verified local alternative or a non-media option. The [fallback hierarchy](/blog/create-no-download-fallback-plan/) defines a safe stop rule.

Record the first trusted reconnection, device order, intended profile, and how progress will be checked before new playback elsewhere.

## Original evidence: red-amber-green audit

| Area | Required evidence | Status | Corrective action |
| --- | --- | --- | --- |
| Schedule | Dependable windows documented | Green / Amber / Red |  |
| Item identity and eligibility | Exact version and current control | Green / Amber / Red |  |
| Disconnected playback | Two samples and tracks tested | Green / Amber / Red |  |
| Storage | Free space and reserve recorded | Green / Amber / Red |  |
| Battery and accessories | Representative test completed | Green / Amber / Red |  |
| Profile and shared ownership | Responsibilities assigned | Green / Amber / Red |  |
| Fallback and reconnection | Sequence documented | Green / Amber / Red |  |

**Green** means current evidence exists. **Amber** means a fix and retest are possible. **Red** means the plan must not rely on that dependency.

## Common mistakes and limitations

- Running the audit minutes before departure.
- Marking a task complete without observable evidence.
- Upgrading amber to green because time is short.
- Treating one episode as proof for a season.
- Ignoring shared-device ownership or accessibility needs.
- Assuming a charging outlet or network will be available.
- Forgetting post-trip reconciliation and cleanup.

## Frequently asked questions

### When should the audit happen?

Run it early enough to replace an item, free storage safely, repeat a battery observation, or choose a different plan. Use the night-before check only as final confirmation.

### Can an amber item remain as a fallback?

No if the fallback itself is expected to be reliable. A useful fallback should pass the same essential tests as the main item.

### Does a green audit guarantee the trip?

No. It documents current evidence for one setup and date. Device, source, account, rights, schedule, and travel conditions can change.
