# Review Your First Norva Trusted-Device Record

> **In short:** Open the current account device view through the official Norva interface and match the new record to a known setup event using broad device type, time window, and administrator. Avoid precise room or person names. Keep only records you recognize and still need; investigate or remove unfamiliar records through the available official controls, then document the outcome without copying secrets.

A device record is useful only when the household can reconcile it with real activity. The first review should happen immediately after onboarding, while the account holder remembers which browser, phone, or television was used. Begin with the [new-account security guide](/blog/secure-new-norva-account/) so account credentials and recovery details are already protected.

## Start from a trusted session

Access the account view from a device and network context you recognize. Navigate through the current official Norva interface rather than a link received in an unexpected message. Confirm the account identity before reviewing or removing anything.

Do not perform the review on a public display. A device list can reveal household technology, timing, or location cues even when it contains no password.

## Reconstruct the expected event

Write a private summary of the onboarding event: approximate local time, broad device category, Norva surface, administrator, and purpose. For a television, use the [safe TV pairing guide](/blog/pair-first-tv-with-norva/) to compare the post-pairing record with the code ceremony.

Avoid recording an IP address, complete user agent, hardware serial, exact room, or full account identifier unless current official support specifically requires a protected diagnostic step.

## Match several signals

Do not trust a friendly device name by itself. Compare the visible time, broad platform, recent action, and your setup log. Some records may use generic browser or operating-system labels, so classify uncertain matches as unknown rather than forcing an identity.

A timestamp can also differ because of time-zone display or delayed activity. Record the interface's displayed time and your local interpretation separately.

## Minimize device names

If the interface allows a custom label, use a short, non-sensitive category such as "Shared TV" or "Personal Phone" only when it is unique enough for the household. Do not include a full name, address, room visible from outside, travel itinerary, workplace, or source name.

A label helps recognition; it does not make a device secure. The underlying session must still be controlled and reviewed.

## Decide: keep, remove, or investigate

Keep a record only when it matches expected activity and the device still needs account access. Remove an obsolete session through the current official control where available. Investigate an unfamiliar record before continuing routine onboarding.

If unfamiliar activity suggests account compromise, follow current official support and security guidance. Change credentials only through the trusted account flow, review other sessions, and preserve a sanitized activity timeline.

## Verify the action

After keeping or removing a record, refresh through the normal interface and confirm the intended state. If a removal signs out the corresponding screen, note that expected operational result. Do not repeatedly click a destructive control because the view seems slow.

The [Norva onboarding journey](/blog/norva-onboarding-complete-journey/) places this review before the first-week audit, when obsolete tests and accidental sessions can be cleaned up again.

Store this register privately. If shared with support, replace device labels with neutral codes and remove precise timestamps or identifiers not required for diagnosis.

## Common trusted-device mistakes

- Assuming a custom label proves identity.
- Ignoring unfamiliar records because the account still works.
- Naming a device with a person, address, or travel plan.
- Removing several sessions before recording the baseline.
- Sharing a complete device-list screenshot publicly.
- Failing to confirm that a removal actually took effect.

## Frequently asked questions

### Is every remembered device automatically safe?

No. Recognition is one signal, not proof. Confirm ownership, current control, expected activity, and whether the session is still needed.

### Should I keep an old device I may use again?

Prefer the least access necessary. If the device is not under current control or no longer needed, remove its session where supported and sign in again through the official flow if circumstances change.

### What if a record is too generic to identify?

Compare time and known activity, preserve the uncertainty, and consult official support if necessary. Do not rename an unknown record merely to make it look familiar.
