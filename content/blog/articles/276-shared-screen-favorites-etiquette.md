# A Simple Favorites Etiquette for Shared Screens

> **In short:** Confirm who is active before saving, ask before changing another person’s preference, and label a choice personal, shared, or temporary. Use a tonight shortlist for group candidates. Remove a favorite only when its owner or agreed shared rule allows it, and verify the action does not alter history or progress.

A shared screen makes actions visible to everyone but does not make every preference collective. Simple etiquette prevents one person’s curiosity from becoming another person’s permanent list.

## Agree on six rules

| Rule | Household agreement |
|---|---|
| 1. Active viewer is named |  |
| 2. Save intent is stated | Personal / shared / temporary |
| 3. Correct destination is used |  |
| 4. Removal authority is clear |  |
| 5. Private detail stays private |  |
| 6. Exceptions get a review date |  |

The agreement should fit on one screen or card. It is a coordination tool, not a record of everyone’s tastes.

## Name the active viewer

Before using a favorite control, read the current account, viewer, or profile label. An avatar alone does not prove isolated state. Use [the profile-scope test](/blog/separate-favorites-by-profile/) with a harmless item before relying on personal separation.

If scope is unknown, say who initiated the save in a minimal private note and avoid bulk changes.

## State the save intent aloud

Use one sentence:

- “I want this in my personal favorites for rewatch.”
- “We want this as a shared long-term reference.”
- “This is only a candidate for tonight.”

The sentence identifies owner, duration, and future action. W3C label guidance supports controls whose purpose and state are explicit; the household language should be equally clear.

## Route temporary choices elsewhere

Do not favorite every suggestion made during group browsing. Use [the tonight shortlist workflow](/blog/create-tonight-shortlist/) with a candidate cap, decision rule, and end-of-evening expiry. The winner gains favorite status only if a durable reason emerges.

## Ask before removing

The person or group that owns the favorite controls removal. An exception is a pre-agreed cleanup rule, such as deleting expired shared candidates after seven days. Before removal:

1. confirm exact title and version;
2. confirm favorite owner and scope;
3. state why the future action ended;
4. remove one item;
5. verify history and progress remain intact.

After watching, use [the post-watch review](/blog/review-favorites-after-watching/) rather than automatic removal.

## Preserve privacy

A shared-screen list need not reveal why someone saved a sensitive title. Use neutral action labels and avoid comments visible to all viewers. Do not display account email, source credentials, or private viewing history during a household review.

DCMI metadata distinctions for title, identifier, type, relation, format, and language can identify the item without personal commentary.

## Handle conflicts

When two people disagree, separate the states. One personal favorite can coexist with a shared-list rejection. When profiles are unavailable or unverified, use minimal personal ledgers and a deliberately shared shortlist.

Norva may retain favorites across supported devices under the same account. Exact profile scope and synchronization should be verified in the current build before the household assumes separation.

## Review the agreement

Revisit the rules after a wrong removal, duplicate-looking entry, profile switch, or new device. Change only the rule that failed. A household does not need continuous auditing when the flow works.

## Use a neutral handoff phrase

Before passing the remote, the active viewer can say: “The current profile is A; I saved one personal item and left the group shortlist unchanged.” The next viewer then confirms the visible label and one known favorite before acting. This short handoff prevents a technically shared screen from becoming an unspoken shared preference space, while revealing profile or scope confusion before another save is made.

## Common mistakes and limitations

- Treating a shared screen as shared preference.
- Saving before checking the active viewer.
- Using favorites for tonight’s entire candidate set.
- Removing another person’s item without consent.
- Writing sensitive preference notes on screen.
- Assuming profile isolation without a test.

## Frequently asked questions

### Can one title be personal and shared?

Yes. Preserve each state only if the interface or fallback record can distinguish its owner and purpose.

### Who cleans expired shared candidates?

Assign an owner or let an agreed expiry rule govern them. Announce the consequence before saving.

### Should children’s preferences be public to the household?

Collect only what the shared decision requires and follow household privacy and supervision rules.
