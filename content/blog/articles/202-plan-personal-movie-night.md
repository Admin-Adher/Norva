# How to Build a Focused Shortlist for Movie Night

> **In short:** Write the session contract first: total time, audience, one or two must-haves, and the current authorized source scope. Generate no more than twelve plausible candidates, verify five against runtime, availability, and required tracks, then advance three preference-ranked finalists. Once three valid choices remain, stop discovery and use a pre-agreed tie-break.

Movie night stalls when discovery and decision happen at the same time. A shortlist funnel gives each activity a boundary: explore broadly, validate narrowly, then choose.

## Write the session contract

Complete this card in one minute:

| Constraint | Decision |
|---|---|
| Available session time |  |
| Audience or profile |  |
| Required audio or subtitles |  |
| Current source scope |  |
| One category preference |  |
| Familiar or new |  |
| Tie-break |  |

Only a requirement should remove a title. A preference should affect ranking unless the group explicitly agrees otherwise.

## Stage 1: generate up to twelve

Start from a broad category, theme, decade, director, favorites list, or recent discovery route. Twelve is a working ceiling, not a universal optimum; use fewer when the library is small.

Capture title, runtime, why it fits, and the exact version currently visible. Do not research every detail yet. The purpose is coverage without endless scrolling.

The Library of Congress facet guidance describes facets as progressive narrowing and warns that records missing a field can disappear. Begin broad enough that incomplete metadata does not define the whole pool.

## Stage 2: validate five

Select five candidates that appear to meet the session contract. For each, verify:

- current availability in the intended source scope;
- runtime plus any planned break;
- required audio and subtitle tracks on the specific version;
- audience suitability according to the household's own rule;
- whether progress or prior viewing changes the decision.

Use [the runtime-based shortlist workflow](/blog/build-runtime-based-movie-shortlist/) when the schedule is the main constraint.

If fewer than five pass, return to the original pool before starting a fresh search.

## Stage 3: rank three finalists

Advance three candidates and give each one sentence:

- **Best fit now:** strongest match to the current energy and time.
- **Safest choice:** familiar or broadly acceptable.
- **Discovery choice:** adds novelty without breaking constraints.

These roles are not permanent labels. They describe the present session and prevent three similar candidates from creating another deadlock.

When guests are involved, use [the guest-browsing preparation guide](/blog/prepare-movie-library-for-guests/) to avoid exposing unrelated profile history or forcing them through the full library.

## Stop discovery at three

Once all three finalists meet the contract, close search and filters. New candidates now need a reason to replace an existing finalist, not simply join the list.

W3C notification guidance recommends clear outcome and recovery messages. Apply that idea manually: state “three verified choices remain” and identify which constraint removed any rejected candidate.

## Use a pre-agreed tie-break

Choose one rule before seeing finalists:

- rotate who decides;
- use a short pairwise comparison;
- select the candidate with the strongest must-have confidence;
- choose the least recently considered option;
- use random selection only if everyone accepts all three.

[The five-to-one decision method](/blog/choose-one-movie-from-five/) provides a structured alternative when the group wants a more deliberate comparison.

## Record only reusable learning

After the choice, note which constraint mattered and whether the selected version matched its metadata. Do not preserve every rejected candidate as a permanent list. Add a genuinely interesting title to favorites or a later queue deliberately.

Norva may group compatible authorized sources, favorites, progress, and version choices in one interface, but current availability and track data still depend on the connected source. Verify the final version before playback.

## Common mistakes and limitations

- Opening filters before agreeing on the session.
- Making preferences into hidden vetoes.
- Adding candidates after the final three are valid.
- Verifying a title but not its specific version.
- Filling the shortlist with near-identical choices.
- Treating the 12–5–3 numbers as mandatory.

The funnel reduces decision cost; it cannot resolve a conflict over the actual must-haves.

## Frequently asked questions

### What if only one movie passes every requirement?

Verify its version and choose it, or revisit the requirements. A shortlist is unnecessary when one valid option remains.

### Should ratings determine the finalists?

Use ratings as one signal, not a substitute for current session constraints or version checks.

### Can the shortlist be saved?

Save it only when the project continues beyond one night. Otherwise preserve the useful favorites, not the temporary elimination state.
