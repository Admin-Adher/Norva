# Capture a TV Guide Issue Without Exposing Private Data

> **In short:** Describe one visible guide symptom, then add masked source and channel samples, short listing cues, device clock and time zone, profile and filters, selected guide date and window, device and application version, source comparison, focus path or layout evidence, timeline, control sample, and every action taken. Remove credentials, full source addresses, complete channel lineups, precise locations, protected descriptions, live-media captures, and unrelated household data before official support submission.

A useful guide packet allows support to reproduce the context without receiving the household's full schedule or source configuration. It should show observations and limits, not a guessed technical cause.

## Write one symptom sentence

Use an observable statement: “Rows appear but program cells are blank on device A,” or “Focus does not move right from sample cell G2.” Avoid “the guide is broken.”

The [TV guide handbook](/blog/tv-guide-troubleshooting-handbook/) identifies the symptom branches.

## Replace identities with stable codes

Use source S1, channels C1–C3, programs P1–P3, device D1, and profile H1. Keep a private mapping only when genuinely necessary. Do not send full source addresses, account emails, channel lists, or credentials.

## Record time context

Include device-local clock, named time zone, UTC offset where visible, automatic or manual state, selected guide date, visible window, and timestamp. Use broad zone labels rather than precise travel locations.

## Record guide context

List masked account, active profile, source label, channel group, favorites or availability view, filters, search query, focused row or cell, and current overlay. Include only controls relevant to the symptom.

## Add minimal channel and listing samples

Use one affected channel, one control, and one adjacent or candidate channel when mapping matters. Record visible identifier cues, short program title cue, start/end times, and source presence. Do not attach complete schedules or descriptions.

The [channel-program crosswalk](/blog/wrong-program-mapped-channel/) provides a safe three-row table.

## Capture device and application state

Record device model, operating system version, Norva application version, network type in generic terms, and foreground or resumed state. Include storage or broader slowness only when relevant and visible.

## Capture focus or layout evidence

For navigation, list starting focus, input, expected target, actual target, and outline state using the [focus trace](/blog/tv-guide-focus-stuck/). For layout, crop a screenshot to the affected rows and obscure unrelated listings. For performance, include the short path from the [scroll sheet](/blog/tv-guide-scroll-performance/).

## Build the timeline

Record source update or confirmation, guide open, refresh request, first symptom, each controlled comparison, application update, device restart, data clearing, source edit, and current state. Include actions even when they appeared ineffective.

## Add one controlled comparison

Change one reversible variable: filter, supported device, trusted network, guide window, or profile only when privacy permits. Record both baselines. State when more than one variable changed.

## Separate facts from hypotheses

Use two fields. “C1 had no program cells at 14:20” is an observation. “The device cache is stale” is a hypothesis unless verified. Do not claim internal mapping, refresh, indexing, or spatial-navigation behavior.

## Redact screenshots and logs

Crop account emails, notifications, household names, unrelated channels, precise location, source endpoints, tokens, and complete schedules. Review diagnostic logs only when support requests them, and remove secrets before submission.

## Avoid protected media capture

Do not record or upload live media to prove a schedule mismatch. A timestamped minimal identity observation is sufficient for the evidence table. Do not reproduce full program descriptions or subtitle text.

## Review destination and retention

Submit Norva interface evidence through official Norva support. Send provider schedule or account evidence to the authorized provider when appropriate. Limit access to the packet and delete temporary copies when no longer needed under applicable guidance.

NIST's Privacy Framework supports managing privacy risk, but it does not replace Norva's current privacy policy or legal advice.

## Common mistakes and limitations

- Sending a broad conclusion instead of one symptom.
- Attaching complete channel lineups or schedules.
- Including credentials, endpoints, or precise locations.
- Capturing protected live media.
- Omitting resets, edits, and repeated refreshes.
- Presenting a hypothesis as product behavior.

## Frequently asked questions

### Can I include a screenshot?

Yes when useful, after cropping and redacting accounts, source details, unrelated listings, notifications, and any secret.

### Should I attach the full channel list?

No. Use a minimal representative sample unless official support specifically requests more through a secure process.

### Can I share my suspected cause?

Label it as a hypothesis and keep it separate from timestamped observations and verified documentation.
