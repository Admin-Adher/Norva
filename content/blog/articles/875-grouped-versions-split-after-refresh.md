# Grouped Versions Split After a Refresh? Compare Identifiers

> **In short:** Freeze the same account, profile, device, application version, source selection, filters, sort, and grouping setting. Preserve the source-refresh timeline and select a few formerly grouped pairs. Compare visible source labels, titles, years, media types, durations, versions, language cues, artwork, metadata, progress, and favorites. Classify changed identity, metadata variation, grouping-context difference, device-specific display, or unknown without guessing Norva's hidden matching logic or deleting cards.

Two cards that used to appear together can separate for several observable reasons, but the interface does not reveal an internal grouping algorithm. Compare what changed and keep implementation claims out of the diagnosis.

## Preserve the before state

Use an earlier redacted screenshot, count, favorite, progress row, or household note to show the cards were previously grouped. Record its timestamp, profile, device, application version, source labels, and grouping setting. If no before evidence exists, describe the current split without presenting prior grouping as certain.

## Rebuild the refresh sequence

Record the source update or refresh, visible sync request and state, first changed card, and first split observation. Include timezone and every repeated request or source edit. Timing matters, but it does not prove causation.

The [import and sync handbook](/blog/catalog-import-sync-troubleshooting-handbook/) keeps the sequence separate from item identity.

## Freeze grouping and filters

Confirm the same Norva account, profile, enabled sources, availability, category, year, rating, audio, subtitles, search query, sort, and grouped-version setting. Toggle no more than one visible control for a documented comparison, then restore the baseline.

## Select representative pairs

Choose up to three split pairs and assign privacy-safe sample codes. Include one control pair that remains grouped if available. Do not export the full catalog. Record which views show the split: category, search, detail, favorite, or progress.

## Compare visible identity cues

For each card, record source label, title, year, media type, season and episode where relevant, duration, resolution or edition cue, audio and subtitle cues, artwork, description, category, and availability. Note punctuation and metadata differences without assuming which field drives grouping.

The [duplicate-after-repeat guide](/blog/duplicates-after-repeat-import/) helps when the cards also look duplicated.

## Check the authorized sources

Through each provider's official route, record whether one or multiple visible versions currently exist and whether their metadata changed. Source observations show available inputs, not how Norva is required to group them.

## Protect progress and favorites

Record progress, completion, and favorite state on both cards. Do not play, merge, delete, mark complete, or refavorite cards during the identity comparison. A split can expose distinct household state that should be preserved.

Use the [progress identity audit](/blog/progress-reset-after-item-identity-change/) if saved position changed.

## Compare one supported device

Use the same trusted account, profile, sources, filters, grouping, and sample on another device close in time. Record both application versions. A split on one device only is a cross-screen observation, not proof of a local cause.

## Interpret catalog counts carefully

Separate cards can change a displayed total depending on its context. Do not infer how Norva counts versions. The [catalog-count guide](/blog/catalog-count-changes-between-imports/) provides the required source, filter, grouping, and sample baseline.

## Avoid forced cleanup

Do not remove and reconnect sources, repeat imports, edit metadata, clear application data, or delete one card to make the view look grouped. Preserve the pair and follow current support guidance. No recovery or regrouping result should be promised.

## Classify the comparison

Use visible identity changed, source membership changed, metadata variation, grouping setting mismatch, filter or view difference, device-specific display, distinct household state, resolved after documented step, or unknown. Keep the classification separate from hypotheses about matching logic.

## Common mistakes and limitations

- Claiming prior grouping without a baseline.
- Comparing different filters, profiles, or grouping settings.
- Guessing which metadata field drives grouping.
- Deleting a card with distinct progress or favorite state.
- Repeating imports or editing source metadata first.
- Treating source versions as a required display outcome.

## Frequently asked questions

### Can I force the cards back into one group?

Do not assume a control or outcome that current Norva guidance does not document. Preserve the pair and ask support.

### Which identifier determines grouping?

Do not infer a hidden identifier. Compare ordinary visible cues and use verified product documentation when available.

### Should I delete the extra-looking card?

Not before checking source version, metadata, progress, favorites, and support guidance. It may represent a legitimate distinct entry.
