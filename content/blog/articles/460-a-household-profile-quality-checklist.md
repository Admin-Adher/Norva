# A Household Profile Quality Checklist

> **In short:** A useful profile has a clear owner and persistent purpose, a distinct privacy-conscious name, separate progress and favourites when needed, explicit language intent, and a predictable shared-screen entry and exit. It is supported by current plan capacity and reviewed with consent. Mark unclear deletion, migration, or privacy questions “defer” rather than guessing.

This checklist evaluates profile structure, not whether someone's viewing choices are good. Use it with profile owners, not as silent monitoring.

## Purpose and ownership

- [ ] The profile has an identified owner or recurring group.
- [ ] Its persistent purpose fits progress, history, favourites, or preferences.
- [ ] It is not named after a device.
- [ ] It is not a forgotten temporary guest context.
- [ ] Shared ownership is understood by everyone affected.

Use [the profile-count planning worksheet](/blog/how-many-household-profiles-do-you-actually-need/) when two contexts overlap.

## Naming and recognition

- [ ] The text label differs clearly from neighbouring profiles.
- [ ] The distinguishing words appear early.
- [ ] The label contains no unnecessary sensitive data.
- [ ] Avatar and colour are supporting clues, not the only identity.
- [ ] The profile is recognisable from the normal TV seat.

## State separation

- [ ] Progress belongs to the intended viewer or group.
- [ ] Favourites have a clear owner.
- [ ] History is not mixed accidentally.
- [ ] Wrong-profile incidents have a recovery record.
- [ ] Shared contexts are intentionally shared.

Norva can preserve these states across supported devices under the same account.

## Language and subtitle intent

- [ ] Preferred audio intent is documented.
- [ ] Subtitle on/off intent is documented.
- [ ] Fallback choices are understood.
- [ ] The household knows tracks depend on source and media.
- [ ] No preference is presented as guaranteed availability.

## Shared-screen routine

- [ ] Every user reads the profile label before play.
- [ ] The TV or tablet is left in an agreed neutral state.
- [ ] Borrowed or untrusted devices are signed out.
- [ ] Notification previews are reviewed appropriately.
- [ ] Profiles are not described as passwords or parental controls.

## Device and plan context

- [ ] Each regularly used device is supported.
- [ ] Devices retaining sign-in are known.
- [ ] Profile capacity matches current official plan details.
- [ ] Capacity is not interpreted as device or concurrency permission.
- [ ] The connected source remains authorised.

## Privacy-respecting review

- [ ] Profile owners participate.
- [ ] Only structural evidence is inspected.
- [ ] Full private histories are not copied.
- [ ] Temporary screenshots and notes have a deletion point.
- [ ] Account emails, credentials, and source URLs are excluded.

Use [the privacy-respecting activity review](/blog/how-to-review-profile-activity-without-invading-privacy/) for consent and minimum evidence.

## Lifecycle readiness

- [ ] New profiles have an onboarding card.
- [ ] Unused profiles have a review trigger.
- [ ] Departing members receive an offboarding review.
- [ ] Removal consequences are verified before activation.
- [ ] Shared state is not deleted without owner agreement.

## Choose one improvement

Do not rename, merge, reset, and offboard simultaneously. Choose the clearest structural issue, make one agreed change, and verify the result.

The [complete household profile guide](/blog/the-complete-guide-to-household-media-profiles/) provides the full profile charter.

## Common mistakes and limitations

Avoid scoring private tastes, forcing every profile into identical preferences, treating capacity as a quality metric, and deleting inactive contexts without review.

Current plan terms and controls can change. This checklist cannot guarantee account security, privacy on another person's device, or state recovery.

## Validate the score with real tasks

Do not approve the profile structure from the scorecard alone. Ask each active household context to complete three ordinary tasks: select the correct profile from the shared screen, start a known item with the intended language settings, and leave the device without exposing the next viewer's state. Record only pass, friction, or blocked.

If one task fails, fix the smallest relevant boundary—name, ordering, handoff rule, or privacy exit—then rerun that task. Rebuilding every profile makes it impossible to know which correction helped and can disturb progress or favourites unnecessarily.

## Frequently asked questions

### Does every profile need to pass every row?

No. Mark irrelevant rows not applicable. Purpose, ownership, naming, privacy, and consequence awareness remain important.

### Is an unused profile low quality?

Not automatically. It may have a future recurring purpose; review ownership and state before deciding.

### Should the account owner run the checklist alone?

Include profile owners whenever their context or private information is affected.
