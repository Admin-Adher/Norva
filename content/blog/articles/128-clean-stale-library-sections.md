# How to Review Stale Sections in a Media Catalog

> **In short:** Call a section stale only when its purpose, membership, source, owner, or retrieval value no longer matches the current collection plan. Low use alone is not proof. Classify each candidate as active, seasonal, archive, overlapping, source-broken, obsolete, or unresolved, then test one reversible action before changing the full structure.

A stale section may be an old category, a source-specific row, a seasonal collection, a migration remnant, or a view no one owns. The review should reveal why it persists.

## Generate candidates from evidence

Use:

- sections with no recent documented retrieval task;
- sections whose item count has not changed while intake rules moved elsewhere;
- source rows tied to an inactive or changed source;
- categories with mostly unavailable items;
- duplicate labels or near-identical membership;
- sections with no owner;
- completed seasonal or project collections;
- migration sections that outlived the cutover.

Do not infer staleness from age alone. Preserve audit date, filters, source, and membership sample.

## Reconstruct the original purpose

Search the collection plan, charter, decision log, migration ledger, and category mappings. Ask:

- Who requested the section?
- Which retrieval task did it support?
- Which source defines it?
- Was it intended to be temporary?
- Does an archive or return trigger exist?
- Which new section may have replaced it?

When no documentation exists, record the gap rather than inventing intent.

## Test current membership

Inspect:

- total visible members with filters cleared;
- unavailable or archived members;
- a sample of unique items;
- overlap with active sections;
- items only reachable through this section;
- source and version context;
- current intake mappings.

Use [the empty category guide](/blog/remove-empty-library-categories/) when the section has no members. Use [the overlapping category matrix](/blog/merge-overlapping-categories/) when membership remains but meaning duplicates another route.

## Classify before acting

- **Active:** still supports an essential task.
- **Seasonal:** useful during a defined period and reviewed afterward.
- **Archive:** infrequent but intentionally retained.
- **Overlapping:** valid items, duplicated purpose.
- **Source-broken:** expected data no longer arrives from the source.
- **Obsolete:** no members, dependencies, or approved purpose.
- **Unresolved:** evidence is insufficient.

The [active-versus-archive framework](/blog/separate-active-and-archive-collections/) prevents low frequency from becoming automatic deletion.

## Choose the smallest useful action

Possible actions:

- update its definition;
- assign an owner and review date;
- relabel it consistently;
- archive it logically;
- merge after an overlap test;
- repair its source mapping;
- remove an obsolete empty structure;
- leave unchanged while evidence is gathered.

Avoid moving or deleting underlying media unless separately authorised and supported. Section cleanup is catalogue work.

## Add the result to routine maintenance

Review temporary and seasonal sections on a defined trigger. Put ownerless, source-broken, and unresolved candidates into named queues. The [monthly cleanup routine](/blog/monthly-catalog-cleanup-routine/) keeps small section drift from becoming another major project.

Record significant removals, merges, or archive changes in [the cleanup documentation guide](/blog/document-catalog-cleanup/).

## Common mistakes and limitations

- Calling every old section stale.
- Using low activity as the only criterion.
- Reviewing with filters active.
- Removing a section that is the only route to unique items.
- Ignoring source-generated behaviour.
- Merging before comparing definitions.
- Treating catalogue removal as permission to delete media.

Usage evidence may be incomplete. Household interviews and retrieval tests should complement counts.

## Frequently asked questions

### How long must a section be unused before review?

There is no universal period. Choose a review trigger appropriate to the section's purpose, such as after a season, project, migration, or several maintenance cycles.

### Should seasonal sections be deleted?

Not automatically. Archive or hide them through supported catalogue controls and define when they should return or be reviewed.

### What if the source recreates a removed section?

Document that it is source-defined and adjust mappings or view rules through supported controls rather than repeatedly deleting it without understanding the source.
