# A One-Page Template for Planning a Media Collection

> **In short:** Keep the one-page plan focused on decisions: purpose, scope, users, authorised sources, essential findability tasks, minimum metadata, active/archive rule, roles, maintenance cadence, pilot, and success checks. Store detailed inventories, mappings, and decision history in linked documents so the plan remains readable during work.

One page is a constraint that forces clarity. It should tell a household what belongs, why the work matters, who may change it, and how success will be checked without becoming an item-level catalogue.

## Complete the template in decision order

### 1. Purpose

Write one sentence naming the people, media, and daily outcome. Avoid “make it organised.” Use a real task such as finding known versions or resuming across supported screens.

### 2. Scope

List included media types, authorised sources, users, languages, and time boundary. Add explicit exclusions and a review date. Use [the scope guide](/blog/define-media-collection-scope/) when the boundary remains ambiguous.

### 3. Household needs

Record each essential viewer or profile context and any device, language, accessibility, or separation requirement.

### 4. Source baseline

List source labels, owners, authorisation status, and high-level contribution. Link to [the full source inventory](/blog/build-media-source-inventory/) without putting credentials on the plan.

### 5. Findability goals

Write three to six tasks with a user, target, starting point, and pass condition. Include at least one edge case such as versions or unavailable items.

### 6. Minimum metadata

Name only the fields needed for those tasks. State the source of truth and unknown-value rule.

### 7. Active and archive rule

Define what stays in daily browsing, what becomes an archive candidate, and how archived material returns.

### 8. Roles and approvals

Name executor, source owner, approver for risky changes, reviewer, and knowledge backup.

### 9. Maintenance

Set intake, regular, seasonal, and event-driven reviews with a time budget.

### 10. Pilot and success

Define the representative sample, rollback, test date, and [success measures](/blog/choose-library-success-measures/).

## Original evidence: copyable one-page plan

```text
COLLECTION PLAN — Version / owner / review date

Purpose:
Scope included:
Scope excluded / review trigger:
Household users and essential needs:
Authorised source labels and owners:
Essential findability tasks:
Minimum metadata and source-of-truth rule:
Active/archive rule:
Executor / approver / reviewer / backup:
Maintenance cadence and time budget:
Pilot sample / rollback / success checks:

Linked evidence:
- Source inventory:
- Metadata rulebook:
- Decision log:
- Migration or cleanup ledger:
```

Complete every line with a decision or “not applicable with reason.” Do not fill unknowns with invented facts.

## Keep detail in linked evidence

The page should not contain:

- passwords, tokens, or recovery codes;
- every media item;
- full metadata mapping tables;
- every past decision;
- private viewing history beyond what the plan needs;
- unsupported compatibility or preservation promises.

Instead, link to controlled working documents. The plan remains the navigation page for the project.

## Test the page with a pilot

Choose ten to twenty items spanning sources, media types, versions, languages, missing data, and archive candidates. Give the plan to the intended maintainer and ask them to:

1. classify the sample;
2. identify required metadata;
3. route exceptions;
4. explain who approves a risky change;
5. run the findability tasks;
6. locate rollback and source documentation.

Record unclear lines. Revise the plan before expanding work.

A successful pilot produces the same classification and escalation route when a second maintainer repeats it independently.

## Version the plan deliberately

At the footer, record version, date, owner, approver, and next review. When the scope or rules change, create a new version and link the reason through [the decision log](/blog/create-library-decision-log/).

Do not change the date merely to make the plan look current. Update it after a material review.

## Common mistakes and limitations

- Turning one page into a full inventory.
- Naming goals with no pass condition.
- Omitting explicit exclusions.
- Including source secrets.
- Assigning every task to the household collectively.
- Defining metadata before retrieval needs.
- Scaling before the pilot reveals exceptions.

A one-page plan guides organisation; it does not verify source rights, guarantee compatibility, or preserve media by itself.

## Frequently asked questions

### Must the template be literally one printed page?

No. The goal is concise decision visibility. A short digital page is acceptable when the essential information remains scannable.

### Where should item-level details go?

Put them in the source inventory, metadata rulebook, exception queue, or migration ledger and link those records from the plan.

### When should the page be updated?

After a material change in scope, source, household need, metadata rule, responsibility, migration, or maintenance cadence.
