# How to Use Filters for a Shared Viewing Decision

> **In short:** Ask each participant for one must-have, one veto, and one preference. Apply only requirements that the group agrees are non-negotiable, then test vetoes without eliminating the whole set. Use preferences to sort or create a shortlist rather than exclude. Stop filtering when three to seven acceptable choices remain, and use a pre-agreed tie-break instead of adding another condition.

Shared browsing becomes slow when every preference is translated into a hard filter. A fair process distinguishes what makes a title unusable from what merely makes it less attractive.

## Write a shared-choice contract

Complete this before opening the filter panel:

| Person | Must-have | Veto | Preference | Reason or accommodation |
|---|---|---|---|---|
| A |  |  |  |  |
| B |  |  |  |  |
| C |  |  |  |  |

Then agree on:

- the current source or catalogue scope;
- a minimum and maximum shortlist size;
- how conflicting requirements will be resolved;
- the tie-break method;
- who confirms track or version details.

The contract makes the decision rule visible. It also prevents the person holding the remote from silently changing the criteria.

## Separate constraints from preferences

A must-have is necessary for someone to participate—for example, a required subtitle language. A veto makes an option unacceptable. A preference influences ranking but does not make other options impossible.

Use this test: **If the condition is absent, can the group still reasonably choose the title?** If yes, it is probably a preference and should influence order or discussion, not membership.

W3C label guidance emphasises controls whose purpose is clear. Use the same discipline in conversation: “English full-dialogue subtitles required” is more actionable than “language.”

## Apply universal requirements first

Start from a documented baseline. Apply the conditions that everyone accepts as requirements, one at a time. Record the count after each transition.

| Step | Agreed requirement | Count before | Count after | Known control passed? |
|---|---|---:|---:|---|
| 1 |  |  |  |  |
| 2 |  |  |  |  |

If one requirement produces zero, clarify whether it is truly universal, whether the relevant metadata is reliable, and whether another available version satisfies it.

The Library of Congress facet guidance notes that records without a filtered field may disappear. Therefore, an empty result may reflect incomplete description as well as genuine incompatibility.

## Add vetoes carefully

Vetoes should be specific and agreed. “No films longer than two hours” is testable when runtime metadata exists. “Nothing boring” is not a reliable filter.

Apply one veto and inspect the set. If it removes most options, return to the contract and decide whether the boundary can be expressed more precisely. Do not compensate by stacking unrelated filters.

## Rank preferences instead of excluding

Use sort order, visible metadata, or a manual shortlist for preferences. [The filtering-versus-sorting guide](/blog/choose-filtering-or-sorting/) explains the distinction: filtering changes who qualifies; sorting changes priority among qualifiers.

Create a shortlist table:

| Candidate | Meets all requirements? | Veto-free? | Preferences met | Version verified? |
|---|---|---|---:|---|
|  |  |  |  |  |

Discuss only candidates that pass the first two columns. This prevents rejected options from repeatedly returning to the debate.

## Use a stop rule

Choose a shortlist range before browsing—three to seven works well as an operational target, not a universal law. Once the set enters that range and all requirements are verified, stop filtering.

If the group keeps adding conditions to avoid choosing, use [the overfiltering stop rule](/blog/avoid-overfiltering-library/). A coin flip, rotation, random selection, or “least recently chosen person decides” can resolve the final tie without distorting requirements.

## Save recurring household constraints

Stable accessibility needs or household rules can become a documented recipe. Task-specific preferences should remain optional. [The repeatable filter-recipe guide](/blog/create-repeatable-filter-recipes/) shows how to record scope and fallback logic.

Norva may retain preferences and organise compatible sources a user is authorised to access across supported devices, but actual tracks, metadata, and current availability depend on the specific source and version. Verify must-haves before starting playback.

## Common mistakes and limitations

- Turning every person's preference into a must-have.
- Using vague vetoes that metadata cannot represent.
- Applying several conditions before checking the count.
- Ignoring the specific available version.
- Letting one profile's saved filters define the group.
- Continuing to filter after a viable shortlist exists.

The contract structures a decision; it cannot resolve value disagreements without an agreed conflict rule.

## Frequently asked questions

### What if two must-haves conflict?

Verify whether alternative versions satisfy both. If not, the group must relax one condition or change the task; filters cannot create a nonexistent match.

### Should ratings be a filter or a sort?

Usually a sort or discussion signal unless the group explicitly agrees on a hard minimum and understands the rating source.

### How many choices should remain?

Pick a range in advance. Three to seven is manageable for many groups, but accessibility and context matter more than a fixed number.
