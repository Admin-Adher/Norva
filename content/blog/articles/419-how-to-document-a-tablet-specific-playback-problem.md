# How to Document a Tablet-Specific Playback Problem

> **In short:** Write one observable symptom, record the tablet and app environment, reproduce it from a known state, and capture the smallest exact step sequence. Compare one controlled condition at a time and remove credentials or private catalogue details from evidence. A useful report distinguishes “tablet-specific” from “only noticed on the tablet.”

“Playback freezes” is a starting symptom, not a diagnosis. The source, item, version, network, audio route, account state, operating system, or app lifecycle can all contribute. Good documentation lets support test the same path without guessing.

## Start with one sentence

Use this pattern:

“On [tablet/environment], when I [single action] with [item/version state], I observe [visible result] instead of [expected result].”

Example structure, without invented data: “After unlocking the tablet on the playback screen, selecting resume leaves the visible timeline unchanged instead of advancing.”

Avoid explanations such as “the player is broken” until evidence isolates the cause.

## Capture the environment

Record only what is useful:

- tablet manufacturer and model as displayed by the device;
- operating-system version;
- Norva app or browser version if visible;
- app or compatible-browser route;
- network type, not private network credentials;
- active profile category, not a person's private name;
- source/version label when safe;
- whether the item is streamed or eligible local media;
- audio output and subtitle state.

Norva works on supported web, mobile, and TV environments. A result on one tablet should not be generalised to every device.

## Reproduce from a known state

1. Pause and return to the item detail screen.
2. Confirm title, episode, and version.
3. Record the approximate position.
4. Perform the minimum steps that cause the symptom.
5. Stop after the first clear failure.
6. Repeat once only if safe and necessary.

Do not clear data, reinstall, reset the tablet, or change several settings before the first evidence capture. Those actions remove useful state and can create new problems.

The [tablet workflow checklist](/blog/a-tablet-viewing-workflow-checklist/) provides a baseline for profile, power, network, audio, and interruption state.

## Separate expected and observed results

Write both in visible terms.

- Weak: “Nothing works.”
- Better expected: “The timeline should advance after I activate the visible resume control.”
- Better observed: “The resume control changes state, but the timeline remains at the same position and no message appears.”

W3C guidance on error identification and status messages explains the value of clear, perceivable feedback. It does not determine the cause of a Norva issue.

## Run one controlled comparison

Change one variable only:

- same item on the tablet after a fresh sign-in;
- different authorised item on the same tablet;
- same item on another supported device;
- same tablet on another trusted network;
- speakers versus headphones;
- streamed versus verified eligible local state.

Do not use the comparison to declare a universal cause. It narrows the condition. For cross-device comparisons, record states with the [handoff state model](/blog/a-state-by-state-guide-to-cross-device-viewing-handoff/).

## Protect the evidence

A screenshot or short recording can help when it shows the control, message, and state transition. Before sharing, redact:

- passwords and recovery codes;
- source credentials or private URLs;
- email addresses and profile names;
- payment or account identifiers;
- unrelated notifications;
- private catalogue information you are not authorised to disclose.

When redaction would remove the useful evidence, describe the state in text instead.

## When to stop troubleshooting

Stop if the tablet shows a security warning, credentials appear unexpectedly, the device overheats or reports a hardware issue, repeated tests alter progress, or the account might be compromised. Follow device guidance or contact support as appropriate.

For routine preparation, use the [complete tablet viewing workflow](/blog/the-complete-guide-to-tablet-viewing-workflows/) rather than turning every session into a diagnostic test.

## Common mistakes and limitations

Avoid vague labels, missing item identity, multiple simultaneous changes, oversized recordings, public links to private evidence, and claiming the tablet is the cause after one observation. Source availability or network state may coincide with the tablet symptom.

A support report improves reproducibility; it cannot guarantee a fix, response time, or diagnosis.

## Frequently asked questions

### Should I send a full screen recording?

Only when necessary and privacy-safe. A short clip or exact text sequence is often easier to review.

### Is one failed item enough to call the issue tablet-specific?

No. Compare another authorised item or supported device while changing one variable. State the result as an observation.

### Should I reinstall before contacting support?

Not before capturing the initial evidence. Reinstallation can remove state and should follow documented support advice.
