# How to Document a Profile-Specific Problem for Support

> **In short:** Record the device, app or browser version, account context, affected profile category, exact item and version when relevant, the state that changed, and the smallest reproducible step sequence. State expected and observed results separately. Compare one consenting profile only if useful, and remove names, emails, credentials, source URLs, full history, and unrelated notifications before contacting support.

A profile problem should be described by state: wrong progress, unexpected favourite, changed language preference, hidden profile label, or wrong startup context. Avoid diagnosing the cause before evidence exists.

Ask the affected profile owner to participate whenever possible. They can confirm the expected state without exposing unrelated history, and they can approve which redacted evidence is safe to share.

## Freeze the first incident

Pause playback and avoid toggling the affected control. Record:

- device role;
- operating-system and Norva version if visible;
- account and anonymised profile label;
- item, episode, and version;
- affected state;
- local date and time zone;
- exact status or error text;
- last successful action.

Do not clear history, switch profiles repeatedly, or reinstall before preserving evidence.

## Write the symptom precisely

Use:

“On [device], under [profile category], after [action], [visible state] appears instead of [expected state].”

W3C error-identification and status-message guidance supports clear feedback, but it does not identify the root cause of a Norva issue.

## Reproduce once

Return to a known profile and item state. Perform the minimum steps needed and stop at the first repeat. Do not use sensitive media if a neutral authorised item can demonstrate the issue.

For a startup problem, follow [the shared-TV wrong-profile diagnostic](/blog/how-to-handle-a-shared-tv-that-opens-the-wrong-profile/) and record the previous exit state.

## Compare carefully

With the other profile owner's consent, change only the profile while keeping device, item, version, and network stable. Record whether the state differs. Do not copy full history or inspect unrelated favourites.

A comparison narrows the condition; it does not prove the profile itself caused the issue.

Record the comparison as “same” or “different” for the affected field only. Do not turn a support test into a broad audit of another household member's activity.

## Original evidence: report template

- **Date, time, time zone:** ___
- **Device and version:** ___
- **Norva route/version:** ___
- **Affected profile category:** ___
- **Affected state:** progress / favourite / preference / startup / other
- **Item/version safe to share:** ___
- **Exact steps:** 1. ___ 2. ___ 3. ___
- **Expected result:** ___
- **Observed result:** ___
- **Visible message:** ___
- **One-variable comparison:** ___
- **Reproducibility:** once / repeated / intermittent
- **Sanitised attachment:** ___

The report is ready only after a human privacy review.

## Redact aggressively

Remove:

- profile names and avatars when identifying;
- account email and payment data;
- passwords and recovery codes;
- source credentials and URLs;
- full history or recommendation screens;
- unrelated notifications.

If redaction destroys the evidence, describe it in text.

Keep attachments short and label source and target screens when several devices are involved. A few annotated observations are easier to review than an untrimmed recording of the entire household interface.

Before submission, ask a second household reviewer to follow the written steps without seeing the original explanation. If they cannot identify the starting profile, affected control, or expected result, revise the report rather than adding more private screenshots.

## Link related device evidence

For a handoff-specific state problem, use [the cross-device failure report](/blog/how-to-document-a-cross-device-handoff-failure/) so source and target are recorded separately.

The [complete household profile guide](/blog/the-complete-guide-to-household-media-profiles/) defines the intended boundaries that support can compare.

## Common mistakes and limitations

Avoid vague wording, several simultaneous changes, unredacted screen recordings, invented timing requirements, and using another person's profile without consent.

A complete report improves reproducibility but cannot guarantee diagnosis, response time, recovery, or transfer of state between profiles.

Retain the case reference and final corrective instruction, then delete temporary private evidence that is no longer needed. Update the household agreement only with the general lesson, not another person's viewing details.

## Frequently asked questions

### Should I include the real profile name?

Usually an anonymised label such as Profile A is enough. Include personal information only when support explicitly needs it through a trusted route.

### Should I test every profile?

No. Start with the affected context and one consenting comparison if it changes only one variable.

### Can I post screenshots publicly?

No. Use the verified support path and share the minimum redacted evidence.
